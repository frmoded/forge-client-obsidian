# Stray `v0.2.6` tag in forge-client-obsidian — minor cleanup candidate

## §1 — What's the message about

While reviewing forge-client-obsidian git state during the v0.2.69 work, I noticed a `v0.2.6` tag in the local tag list — sitting between `v0.2.59` and `v0.2.60` in lexicographic-then-version order. Almost certainly a truncation artifact from a historical `git tag v0.2.60` typo (someone typed `git tag v0.2.6` then re-tagged correctly as `v0.2.60`, but the typo'd tag wasn't deleted).

Reproduce:

```
cd ~/projects/forge-client-obsidian && git tag --list 'v0.2.*' | sort -V | head -20
```

Expected to see `v0.2.6` between `v0.2.59` and `v0.2.60`.

## §2 — Why it matters (minor)

Mostly harmless, but potentially trips install scripts or release tooling that iterate tags by version-string parsing. The forge-installer's BRAT path uses GitHub's release API, not git tags directly, so it shouldn't be affected — but `install-latest.sh` and any future tooling that walks tags via `git tag` could pick `v0.2.6` and try to parse it as either v0.2.6 (treating the trailing `6` as the patch) or as malformed.

No active failure mode I've observed. Filing as cleanup candidate, not urgent.

## §3 — Suggested fix

```
cd ~/projects/forge-client-obsidian && git tag -d v0.2.6 && git push --delete origin v0.2.6
```

One-line drain prompt would do it; or fold into the next plugin release prompt as a §0 step.

## §4 — Context

- No GH release exists for `v0.2.6` (the actual `0.2.6` release is `v0.2.60`).
- Tag itself doesn't point at anything load-bearing — likely a non-annotated tag on some old commit from the truncation moment.
- Cohort install path (BRAT-via-forge-installer) is unaffected per §2.

Driver: no urgency. Fold into next plugin work or stand-alone cleanup drain when convenient.
