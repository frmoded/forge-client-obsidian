# v0.2.22 — wire `inputs` through computeViaEngine to fix Greet kwargs drop

## Why this prompt exists

Production smoke against v0.2.21 surfaced the actual Greet TypeError
root cause. It is NOT a staleness race (which v0.2.17/v0.2.18/v0.2.19/
v0.2.21 progressively chased). It is a hardcoded-empty-dict bug at
the JS→Python compute boundary that has been latent since v0.2.6.

Trace of the bug:

1. User edits Greet.md to `def compute(context, name): print("hello " + name)`,
   frontmatter `inputs:` empty.
2. Click Forge. `runSnippet` calls `host.getInputNames("Greet")` →
   returns `["name"]` (v0.2.20+v0.2.21 work here).
3. Modal opens with `name` field. User types "Foo". Click Run.
4. Modal submit fires `onRun({name: "Foo"}, ...)`.
5. `computeSnippetWithArgs(vaultPath, "Greet", [], {name: "Foo"})` ✓
6. `computeSnippet(serverUrl, vaultPath, "Greet", [], {name: "Foo"})` ✓
   (signature at `server.ts:198`: `(serverUrl, vaultPath, snippetId, args, inputs)`)
7. **`server.ts:220`: `host.computeViaEngine(snippetId, args)`** — the
   `inputs` argument is silently dropped. computeViaEngine's JS-side
   signature is `(snippet_id, args)`, no `inputs` param.
8. Python `_forge_compute(snippet_id, args, inputs, vault_name)` runs
   with hardcoded `inputs={}` (literal at `pyodide-host.ts:831`):
   ```python
   _forge_compute(_forge_snippet_id, list(_forge_args_in or []), {}, _forge_vault_name)
   ```
9. `exec_python` calls `fn(context, *args, **{})` against
   `def compute(context, name)` → TypeError missing 'name'.

Fix is 4 lines across two files:

- `pyodide-host.ts:775` — extend `PyodideHostInstance.computeViaEngine`
  interface to accept `inputs: Record<string, unknown>`.
- `pyodide-host.ts:821` — extend impl signature to accept `inputs`,
  set as global, pass to Python.
- `pyodide-host.ts:831` — replace hardcoded `{}` with the global.
- `server.ts:220` — pass `inputs` through:
  `host.computeViaEngine(snippetId, args, inputs)`.

The Python `_forge_compute` already accepts `inputs` and forwards
correctly via `exec_python(code, ..., inputs=inputs, ...)`. No engine
change.

## 2. Fix shape

### 2.1 `pyodide-host.ts` interface and impl

Line 775 — extend interface:

```typescript
// BEFORE
computeViaEngine(snippet_id: string, args: unknown[]): Promise<ComputeResult>;

// AFTER
computeViaEngine(
  snippet_id: string,
  args: unknown[],
  inputs?: Record<string, unknown>,
): Promise<ComputeResult>;
```

`inputs` is optional with default `{}` so existing callers that
don't pass kwargs (the moda fast-path, iframe protocol) still work
unchanged.

Line 821 — extend impl:

```typescript
async computeViaEngine(
  snippet_id: string,
  args: unknown[],
  inputs: Record<string, unknown> = {},
): Promise<ComputeResult> {
  this.pyodide.globals.set("_forge_args_in", args as any);
  this.pyodide.globals.set("_forge_snippet_id", snippet_id);
  this.pyodide.globals.set("_forge_inputs_in", inputs as any);  // NEW
  this.pyodide.globals.set("_forge_vault_name", "");
  const tuple = this.pyodide.runPython(`
_forge_compute(_forge_snippet_id, list(_forge_args_in or []), dict(_forge_inputs_in or {}), _forge_vault_name)
`);
  // ... rest unchanged
}
```

Note the `dict(_forge_inputs_in or {})` cast — JS objects coming
through `globals.set` arrive as PyProxy / Map-like; the `dict()`
constructor coerces to a plain dict that `**kwargs` unpacking
accepts. Without it, `**` against a PyProxy fails.

### 2.2 `server.ts:220` call site

```typescript
// BEFORE
const out = await host.computeViaEngine(snippetId, args);

// AFTER
const out = await host.computeViaEngine(snippetId, args, inputs);
```

The `inputs` is already the function parameter at line 203:
`computeSnippet(serverUrl, vaultPath, snippetId, args, inputs)`.
Just pass it through.

## 9. One observation worth flagging

This bug is the canonical example of WHY the new "push every
assertion to suite up to UI boundary" protocol matters. The race
chase (v0.2.17 → v0.2.21) consumed five releases looking for the
wrong root cause. A suite-level test that called
`host.computeViaEngine('any-snippet', [], {x: 1})` and asserted x
reached Python would have caught the latent v0.2.6 bug immediately.

For the v1.0 retrospective: every JS→Python boundary helper
(`computeViaEngine`, `syncUserVaultFile`, `getInputNames`,
`preflightThenInventory`, `getConnectInventory`, etc.) should
have at least one test that round-trips a non-trivial argument
through Python and asserts it arrived intact. The boundary itself
is a class of bug that unit tests of either side miss.
