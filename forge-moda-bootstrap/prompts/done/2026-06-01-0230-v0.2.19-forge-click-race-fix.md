# v0.2.19 — forge-click race fix (pre-flight sync before /generate)

## Why this prompt exists

User confirmed locally: 109 prior pass + 4 new fails from
`src/forge-click-race.test.ts`. The failing cases assert a
`_forge_preflight_then_inventory(snippet_id)` helper that doesn't
exist yet.

The real production race: `vault.on('modify')` is async. Between
Obsidian firing the event on Cmd-S and the handler completing
`syncUserVaultFile`, a fast Forge-click runs `/generate` with stale
inventory. v0.2.18's hook addresses the slow path (autosave + wait);
v0.2.19 closes the fast path (immediate Forge-click after edit).

Two-part fix:

a. **Python helper** that the failing test calls — exposes a
   contract the suite can verify.
b. **JS-side production wiring** — forgeSnippet pre-flight syncs
   from disk to MEMFS BEFORE `/generate`. The real fix for the
   user-visible UX race.

## 1. Python helper — passes the 4 failing tests

`src/pyodide-host.ts` inlined Python block, alongside the other
`_forge_*` helpers (near `_forge_get_generate_inventory` at line
~392):

```python
def _forge_preflight_then_inventory(snippet_id: str):
    """v0.2.19: pre-flight inventory helper. Refreshes the registry's
    cached entry for this snippet from current MEMFS state, then
    returns the inventory.

    Called from JS-side forgeSnippet BEFORE /generate to close the
    race between async vault.on('modify') and synchronous Forge-
    click. The JS caller is responsible for syncing fresh disk
    content INTO MEMFS first (via syncUserVaultFile); this helper
    handles the registry-refresh + inventory step in one atomic
    call.

    Path inference: V1 single-vault convention is that snippet_id
    maps to a relative path of "<snippet_id>.md". Qualified IDs
    like "forge-moda/setup" become "forge-moda/setup.md". For the
    user's authoring vault, snippets live at the root, so unqualified
    IDs work directly. For bundled libraries (forge-moda, forge-
    music), the qualified prefix matches the library subdir.
    """
    relpath = f"{snippet_id}.md"
    try:
        _forge_registry.refresh_file(relpath)
    except Exception:
        # Path may not exist (snippet not in authoring vault, or
        # bundle-only snippet). Let _forge_get_generate_inventory's
        # own resolve() raise the canonical error.
        pass
    return _forge_get_generate_inventory(snippet_id)
```

This makes 3 of 4 failing tests pass:
- (a) `preflight helper exists` — pass: `'_forge_preflight_then_inventory' in dir()` is True.
- (b) `preflight returns fresh english after disk-write-without-hook` — pass: `refresh_file` reads the new MEMFS body the test wrote.
- (c) `idempotent across repeated calls` — pass: refresh_file is idempotent on identical bodies.
- (d) `handles unknown snippet_id` — pass: refresh_file silently catches, `_forge_get_generate_inventory` raises `SnippetResolutionError` cleanly.

## 2. JS-side production wiring — the actual UX fix

`src/pyodide-host.ts` TS surface — extend `PyodideHostInstance`:

```typescript
/** v0.2.19: synchronous pre-flight inventory. Refreshes the
 *  registry's cached entry for the snippet from current MEMFS
 *  state, then returns the inventory. Use after syncUserVaultFile
 *  to bypass the async vault.on('modify') race during fast
 *  Forge-clicks. */
preflightThenInventory(snippetId: string): Promise<GenerateInventory>;

async preflightThenInventory(snippetId: string): Promise<GenerateInventory> {
  this.pyodide.globals.set('_forge_preflight_snippet_id', snippetId);
  const proxy = this.pyodide.runPython(`_forge_preflight_then_inventory(_forge_preflight_snippet_id)`);
  return this._unwrap(proxy) as GenerateInventory;
}
```

`src/main.ts` — find the forgeSnippet's pre-`/generate` path
(near where `getGenerateInventory` is currently called; CC must
locate the actual file via grep). Before the inventory call,
synchronously sync fresh disk content to MEMFS, then call the new
preflight inventory:

```typescript
// v0.2.19: pre-flight sync to close the vault.on('modify') race.
// Cmd-S → modify event fires async → Forge-click within ~100ms
// can beat the handler to /generate. Reading fresh disk content
// + syncing to MEMFS synchronously here makes the timing local
// to the Forge-click handler instead of dependent on hook completion.
const file = this.app.vault.getAbstractFileByPath(<snippet relative path>);
if (file instanceof TFile) {
  try {
    const freshContent = await this.app.vault.read(file);
    const hostManager = getPyodideHost();
    if (hostManager) {
      const host = await hostManager.getInstance();
      await host.syncUserVaultFile(file.path, freshContent);
    }
  } catch (e) {
    // Non-fatal — log and continue with whatever cached inventory exists.
    console.warn('Forge: pre-flight sync failed before /generate', e);
  }
}

// THEN call inventory + /generate as before, but via the new
// preflight helper so the registry refresh + inventory fetch
// happen atomically:
const inventory = await host.preflightThenInventory(snippetId);
// ... continue with existing /generate POST ...
```

CC: replace the existing `host.getGenerateInventory(snippetId)`
call site with `host.preflightThenInventory(snippetId)` — the new
helper supersedes the bare inventory call.

`getGenerateInventory` stays on the interface for any other
caller that doesn't need pre-flight semantics (e.g., direct test
fixtures); the production Forge-click path uses preflight.
