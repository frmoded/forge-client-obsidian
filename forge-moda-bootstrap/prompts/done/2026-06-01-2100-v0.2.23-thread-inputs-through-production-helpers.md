# v0.2.23 — thread `inputs` through production Python helpers + harden fixture against drift

## Why this prompt exists

v0.2.22 fixed the JS side of the kwargs-drop bug. Production smoke
against v0.2.22 STILL fires `TypeError: compute() missing 1 required
positional argument: 'name'`. Triage (user expanded `inputs:{…}` in
the dev console): kwargs `{name: "world"}` IS reaching
`computeSnippetWithArgs` and is correctly threaded into the JS-side
`computeViaEngine(snippetId, args, inputs)` call.

The bug is downstream — in the **production Python helpers**:

`src/pyodide-host.ts:587` (the inline Python block):

```python
def _forge_run_snippet(snippet_id: str, args, vault_name=None):
    """..."""
    reg, resolver = _forge_get_resolver(vault_name)
    snip = resolver.resolve(snippet_id)
    snippet_type = snip.get("meta", {}).get("type")
    if snippet_type == "action":
        code = extract_python(snip["body"])
        ...
        stdout, result = exec_python(
            code, {},                     # ← {} HARDCODED — inputs lost
            resolver, args=tuple(args),
            vault_path=_forge_user_vault,
            registry=reg, snippet_id=snip["snippet_id"],
        )
```

`src/pyodide-host.ts:624`:

```python
def _forge_compute(snippet_id: str, args, inputs, vault_name: str):
    """..."""
    _ = inputs                            # ← DISCARDED
    reg, resolver = _forge_get_resolver(vault_name)
    snip = resolver.resolve(snippet_id)
    stdout, raw_result = _forge_run_snippet(snippet_id, args, vault_name)   # ← no inputs param
    result = serialize_result(raw_result, snip)
    return result, stdout
```

Even though v0.2.22's JS-side `_forge_compute(_forge_snippet_id, list(_forge_args_in or []), _forge_inputs_in.to_py() if _forge_inputs_in else {}, _forge_vault_name)` correctly passes a Python dict for `inputs`, the production `_forge_compute` immediately discards it via `_ = inputs` and calls `_forge_run_snippet` without it; `_forge_run_snippet` then hardcodes `{}` to `exec_python`.

**CC's v0.2.22 tests passed because the test fixture defined its
OWN versions of `_forge_run_snippet` and `_forge_compute` that DID
thread inputs correctly.** The fixture at
`src/compute-kwargs.test.ts:131-149` claims "VERBATIM mirror of
production" via a drift-protection NOTE — but it's not verbatim:
the test version's `_forge_run_snippet` has a 4-arg signature
`(snippet_id, args, inputs, vault_name=None)` and threads inputs
into `exec_python(code, inputs, ...)`. Production has a 3-arg
signature `(snippet_id, args, vault_name=None)` and uses `{}`.

The test fixture pre-applied the fix; the production code didn't
get updated. Suite passes against fixture; production breaks. This
is exactly the failure mode the new "test fixtures that mirror
production must dynamically load or static-check for drift"
protocol rider was just added for (cc-prompt-queue.md).

## 1. TDD discipline (HARD RULE) — failing test against PRODUCTION

The v0.2.22 tests are misleading because they validate the fixture
shape, not the production shape. Restructure first; then write
failing tests that exercise the production code path.

### 1.1 Restructure the test fixture to load production Python dynamically

`src/compute-kwargs.test.ts:bootGreet` — replace the inline `py.runPython(...)` block that defines `_forge_run_snippet` and `_forge_compute` with one that loads them from `src/pyodide-host.ts`:

// v0.2.23: load the production inline Python block dynamically so the
// test exercises production code, not a drift-prone mirror. The block
// in pyodide-host.ts is bounded by two grep-able markers; extract
// everything between them and run it.
const hostSource = fs.readFileSync(
  path.resolve(process.cwd(), 'src/pyodide-host.ts'),
  'utf-8',
);

// Find the inline Python block. The block is a runPython template
// literal. Use the existing marker comments (or add explicit ones).
// Conservative regex: pull everything between `_PYTHON_BLOCK_BEGIN`
// and `_PYTHON_BLOCK_END` markers we'll add as part of this prompt.
const pythonBlockMatch = hostSource.match(
  /\/\/ _PYTHON_BLOCK_BEGIN\n([\s\S]*?)\/\/ _PYTHON_BLOCK_END/,
);
if (!pythonBlockMatch) {
  throw new Error('Could not extract _PYTHON_BLOCK from pyodide-host.ts');
}
const productionPython = pythonBlockMatch[1]
  // Strip line-ending JS backslashes if any
  .replace(/\\\\/g, '\\')
  .replace(/\\n/g, '\n');

py.runPython(productionPython);

### 1.3 Apply the production fix

Two changes to the inline Python block in `src/pyodide-host.ts`:

**a. `_forge_run_snippet` accepts `inputs`:**

```python
def _forge_run_snippet(snippet_id: str, args, inputs=None, vault_name=None):
    """Run a snippet and return (stdout, result). ..."""
    if inputs is None:
        inputs = {}
    reg, resolver = _forge_get_resolver(vault_name)
    snip = resolver.resolve(snippet_id)
    snippet_type = snip.get("meta", {}).get("type")
    if snippet_type == "action":
        code = extract_python(snip["body"])
        # ... diagnostic print unchanged ...
        stdout, result = exec_python(
            code, inputs,                # ← was {}, now inputs
            resolver, args=tuple(args),
            vault_path=_forge_user_vault,
            registry=reg, snippet_id=snip["snippet_id"],
        )
    elif snippet_type in ("data", "snapshot"):
        from forge.core.executor import read_data_snippet
        result = read_data_snippet(snip)
        stdout = ""
    else:
        raise ValueError(f"unknown snippet type '{snippet_type}' for {snippet_id!r}")
    return stdout, result
```

The `inputs=None` default keeps existing callers compatible (moda
fast-path, iframe protocol pass only positional args).

**b. `_forge_compute` passes inputs through:**

```python
def _forge_compute(snippet_id: str, args, inputs, vault_name: str):
    """Phase 1 entry point — kept for backward compat. ..."""
    reg, resolver = _forge_get_resolver(vault_name)
    snip = resolver.resolve(snippet_id)
    stdout, raw_result = _forge_run_snippet(snippet_id, args, inputs, vault_name)  # ← inputs threaded
    result = serialize_result(raw_result, snip)
    return result, stdout
```

Drop the `_ = inputs` discard line. The misleading "we forward
through" comment now matches actual behavior.

## 8. One observation for the v1.0 retrospective

This is the third "JS→Python boundary" bug shape now:
- v0.2.22: JS-side `computeViaEngine` dropped inputs.
- v0.2.23: Python-side `_forge_compute` + `_forge_run_snippet` drop inputs.

Both bugs were latent since v0.2.6 (Pyodide compute path
introduction). Both were masked by test fixtures that pre-applied
the fix shape. The lesson for v1.0 audit:

**Every test that claims to "mirror production" must verify the
claim mechanically.** Dynamic loading from source is the cheapest
mechanism; static drift-checks (string comparison against named
lines in the source file at test-start) are acceptable. Comments
naming a drift-protection convention are insufficient — they don't
gate the build.

The 6-release plumbing arc (v0.2.17 → v0.2.23 once it ships) all
trace to one fundamental boundary that wasn't tested against the
real code path. The v1.0 audit item (m) for "every JS→Python
boundary helper needs a round-trip test for non-trivial arguments"
is incomplete without (m'): "and the test must call into production
code via dynamic load, not a mirror that can drift."
