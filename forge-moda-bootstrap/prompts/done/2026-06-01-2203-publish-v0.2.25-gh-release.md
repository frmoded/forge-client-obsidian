# Publish v0.2.25 GitHub Release so install-latest.sh picks up the blues bundle

## Scope

Publish the existing v0.2.25 tag of `forge-client-obsidian` as a GitHub
Release with the release zip attached, so `scripts/install-latest.sh`
(which queries the GH `releases/latest` endpoint) actually downloads
v0.2.25 instead of the prior v0.2.24. The plugin code, manifest bump,
commit, tag, and push for v0.2.25 are already on `origin/main` — only
the GH Release artifact is missing.

What this prompt does NOT do:
- Bump the manifest version (already at 0.2.25).
- Create new commits or tags (already exist).
- Modify any source code.
- Update the `build-release-zip.mjs` preflight to include
  `assets/vaults/forge-music/blues/` in `REQUIRED_FILES` (separate
  follow-up; flag it).
- Touch any vault other than `forge-client-obsidian`.

## Why

Yesterday's prompt `2026-06-01-2140-blues-backup-into-forge-music`
bundled the blues content into the plugin at
`assets/vaults/forge-music/blues/` and tagged the result as v0.2.25
on `origin/main`. The tag exists; the commit exists; the bundle on
disk is correct. But the user's install path —
`VAULT=~/forge-vaults/test1 bash scripts/install-latest.sh` — pulls
from the latest GitHub Release, and the most recent published GH
Release is v0.2.24. So the user's test install came up at v0.2.24
without the blues bundle, blocking the user-side smoke for the
prior prompt.

This prompt closes that gap: build the v0.2.25 release zip, publish
the GH Release with the zip attached, and verify that
`install-latest.sh` now resolves to v0.2.25 and that a fresh
install carries the blues subdirectory through.

- A new GitHub Release on `frmoded/forge-client-obsidian` tagged

No source-tree files are modified.

## Implementation notes

1. **Verify the v0.2.25 commit + tag are already on origin/main.**
   ```
   cd ~/projects/forge-client-obsidian
   git log --oneline -3
   git tag --list 'v0.2.25'
   git ls-remote origin refs/tags/v0.2.25
   ```
   Expected: HEAD or one of the recent commits is
   `a2cf544 [2026-06-01-2140-blues-backup-into-forge-music] v0.2.25 — bundle forge-music/blues/ subdirectory (forge-music v0.3.1)`,
   the tag `v0.2.25` exists locally, and `ls-remote` shows the tag on
   origin. If any of these is missing, route to `failed/` with a
   message naming the missing piece.

2. **Verify no v0.2.25 GH Release exists yet.**
   ```
   gh release view v0.2.25 --repo frmoded/forge-client-obsidian
   ```
   Expected: exits non-zero with "release not found" (or equivalent).
   If a release ALREADY exists, route to `questions/` —
   re-publishing would require deleting the existing release first,
   which the user should authorize explicitly.

3. **Verify manifest.json is at 0.2.25.**
   ```
   jq -r '.version' ~/projects/forge-client-obsidian/manifest.json
   ```
   Expected: `0.2.25`. If not, route to `failed/` — the tag and
   manifest don't agree, which is a deeper state problem to
   investigate before proceeding.

4. **Build the release zip.**
   ```
   cd ~/projects/forge-client-obsidian
   npm run build
   npm run release-zip
   ```
   Expected: `dist/forge-client-obsidian-v0.2.25.zip` produced.
   Report path, size in MB, and SHA-256 (the script prints all
   three).

5. **Verify the zip contains the blues bundle.**
   ```
   unzip -l dist/forge-client-obsidian-v0.2.25.zip \
     | grep -E '(forge-music/(blues/|forge\.toml))' \
     | sort
   ```
   Expected: 9 lines —
   `forge-client-obsidian/assets/vaults/forge-music/forge.toml` plus
   8 blues `.md` files (chorus, form, guitar_solo_chorus,
   solo_chorus, song, twelve_bar_blues_progression, vocal_phrase_a,
   vocal_phrase_b). If `blues/` lines are missing, route to
   `failed/` — the bundle on disk doesn't match what got zipped.

6. **Clean-vault smoke (per cc-prompt-queue release-shipping rule).**
   Set up a fresh test directory; download/unzip the artifact as
   `install-latest.sh` would; verify the structure:
   ```
   SMOKE_DIR=/tmp/forge-v0.2.25-smoke
   rm -rf "$SMOKE_DIR" && mkdir -p "$SMOKE_DIR/.obsidian/plugins"
   unzip -q ~/projects/forge-client-obsidian/dist/forge-client-obsidian-v0.2.25.zip \
     -d "$SMOKE_DIR/.obsidian/plugins/"
   ls "$SMOKE_DIR/.obsidian/plugins/forge-client-obsidian/"
   ls "$SMOKE_DIR/.obsidian/plugins/forge-client-obsidian/assets/vaults/forge-music/"
   ls "$SMOKE_DIR/.obsidian/plugins/forge-client-obsidian/assets/vaults/forge-music/blues/"
   jq -r '.version' "$SMOKE_DIR/.obsidian/plugins/forge-client-obsidian/manifest.json"
   ```
   Expected:
   - Top-level plugin dir contains `main.js`, `manifest.json`, `assets/` at minimum.
   - `assets/vaults/forge-music/` contains `blues/` subdir plus `forge.toml`, `form.md`, `twelve_bar_blues_progression.md`.
   - `assets/vaults/forge-music/blues/` contains 8 `.md` files.
   - manifest version is `0.2.25`.

   Capture all four output blocks verbatim in feedback.

7. **Publish the GH Release.**
   ```
   cd ~/projects/forge-client-obsidian
   gh release create v0.2.25 \
     --repo frmoded/forge-client-obsidian \
     --title "v0.2.25 — bundle forge-music/blues/ subdirectory" \
     --notes "Bundles forge-music v0.3.1 including the blues/ subdirectory (8 snippets). When a user vault declares domains = [\"music\"], ensureBundledForgeMusic extracts forge-music/blues/ into the vault root via the recursive copyDirRecursive helper. No code changes since v0.2.24 — release-only bump to ship the bundled content." \
     dist/forge-client-obsidian-v0.2.25.zip
   ```
   Capture the release URL printed by `gh release create`.

8. **Verify install-latest.sh now resolves to v0.2.25.**
   ```
   curl -fsSL https://api.github.com/repos/frmoded/forge-client-obsidian/releases/latest \
     | grep '"tag_name"' | head -n1
   ```
   Expected: `"tag_name": "v0.2.25",`. If the API still returns
   v0.2.24, GH may take a few seconds to propagate — retry once
   after `sleep 5`. If it still returns the old version after the
   retry, capture the response body and route to `failed/` —
   something about the release flagging may have gone sideways
   (draft vs published, target branch mismatch, etc.).

9. **Verify SHA-256 round-trip.**
   ```
   curl -fsSL https://api.github.com/repos/frmoded/forge-client-obsidian/releases/tags/v0.2.25 \
     | grep -A1 '"name": "forge-client-obsidian-v0.2.25.zip"' \
     | grep '"digest"' | head -n1
   ```
   Expected: the digest line includes the same SHA-256 that step 4
   reported for the local zip. If they don't match, the upload may
   be corrupted — capture both and route to `failed/`.

10. **No new commits.** The work in this prompt produces only the
    GH Release artifact (and the local `dist/` zip, which is
    gitignored). No `git commit`, `git push`, or `git tag`
    operations needed beyond what's already done. The default-on
    git policy doesn't apply here.

## Don'ts

- **Don't run `release.sh`.** It would attempt to bump manifest
  again (failing on "version equals current"), and even if it
  passed it would ship only the BRAT assets — not the zip the
  install path needs.
- **Don't bump manifest.json.** Version is already at 0.2.25.
- **Don't create new commits or tags.** This is a release-only
  artifact publication.
- **Don't delete any existing GH Release.** If one already exists
  for v0.2.25, route to `questions/`.
- **Don't push or rebase or modify git history.** Read-only on git
  state for this prompt.
- **Don't proceed past §2 if a v0.2.25 GH Release already exists.**
  Route to `questions/` so the user decides whether to delete +
  republish or accept the existing one.
- **Don't skip the clean-vault smoke.** It's the load-bearing
  check that catches "the zip uploaded but is broken" before the
  user re-runs install-latest.sh.
