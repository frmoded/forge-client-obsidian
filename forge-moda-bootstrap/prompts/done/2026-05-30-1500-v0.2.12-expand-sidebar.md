# v0.2.12 hotfix — expand collapsed sidebar before revealLeaf

## Why this prompt exists

v0.2.11 added `revealLeaf(existing)` to `getOutputView`'s existing-leaf
branch so a Forge-click would bring the output panel into view. Smoke
caught a residual case: if the right sidebar is collapsed,
`revealLeaf` activates the tab inside the sidebar but doesn't expand
the sidebar itself. The user sees the snippet run (console logs
confirm compute + render), but visually nothing changes.

Verified via dev console: with the sidebar collapsed and an existing
forge-output leaf, manually clicking the `>>` icon to expand the
sidebar reveals the correct output, confirming the leaf and its
render are healthy — only the sidebar's collapsed state isn't being
flipped by `revealLeaf` in some Obsidian versions.

## 1. Patch

`src/main.ts` `getOutputView`, existing-leaf branch (current state
from v0.2.11):

```typescript
const existing = this.app.workspace.getLeavesOfType(OUTPUT_VIEW_TYPE)[0];
if (existing) {
  // v0.2.11: also reveal when the leaf already exists. ...
  this.app.workspace.revealLeaf(existing);
  return waitForRealView(existing);
}
```

Change to:

```typescript
const existing = this.app.workspace.getLeavesOfType(OUTPUT_VIEW_TYPE)[0];
if (existing) {
  // v0.2.12: expand the containing sidebar first if collapsed.
  // revealLeaf alone activates the tab but does NOT unfold a
  // collapsed WorkspaceSplit in some Obsidian versions — the leaf
  // stays offscreen and the user sees nothing happen.
  const root = existing.getRoot() as unknown as { collapsed?: boolean; expand?: () => void };
  if (root?.collapsed && typeof root.expand === 'function') {
    root.expand();
  }
  this.app.workspace.revealLeaf(existing);
  return waitForRealView(existing);
}
```

The cast is necessary because Obsidian's public type for `getRoot()`
returns `WorkspaceItem`, which doesn't surface `collapsed`/`expand`
in the type defs — but `WorkspaceSplit` (the actual runtime type for
sidebar roots) does have both. Belt-and-suspenders `typeof === 'function'`
check covers the edge case where the leaf's root isn't a sidebar at
all (e.g., it's in the main rootSplit because a user dragged it
there); in that case `.expand` is undefined and we silently skip
without erroring.

Apply the same expand-before-reveal pattern to the new-leaf branch
if it makes sense (the existing branch creates a new leaf via
`getRightLeaf(false)` which already targets the sidebar; setViewState
should auto-expand, but check — if a future Obsidian version
regresses that behavior, the parity is cheap).

## 4. Manual smoke guidance for user

1. Install v0.2.12 (corrected unzip path).
2. Reload Obsidian.
3. **Collapse the right sidebar** explicitly: click the `>>` icon on
   the right edge of the window to fold it away. Confirm via
   screenshot or visual check that the sidebar is gone.
4. Open a forge-moda snippet (e.g., `forge-moda/setup.md`).
5. Forge-click.
6. Expect: right sidebar **expands automatically**, Forge Output tab
   activates, snippet result renders.

If steps 5-6 work → v0.2.12 ships.

If the sidebar stays collapsed → diagnostic from session:

```javascript
const l = app.workspace.getLeavesOfType('forge-output')[0];
console.log({
  root: l?.getRoot()?.constructor?.name,
  rootCollapsed: (l?.getRoot() as any)?.collapsed,
  hasExpand: typeof (l?.getRoot() as any)?.expand,
});
```

Paste output if it fails — most likely cause is `expand` is named
differently in the user's Obsidian version or the leaf's root isn't
the sidebar.
