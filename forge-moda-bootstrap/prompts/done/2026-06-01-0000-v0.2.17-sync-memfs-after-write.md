# v0.2.17 — sync MEMFS after writeGeneratedCode (fix greet-snippet staleness)

## Why this prompt exists

v0.2.16's diagnostic confirmed: when the user clicks Forge on a
snippet:

1. `/generate` succeeds, returns new Python.
2. `writeGeneratedCode` writes new Python to disk via
   `app.vault.modify`.
3. Pyodide compute then runs — but reads from MEMFS, which was
   snapshotted at plugin init. **Stale.** The OLD Python runs.

Disk and MEMFS diverge. User sees:

- Editor shows the new Python (Obsidian reads from disk).
- Forge Output shows the old computation result.
- `console: Forge debug: run_snippet('Greet') body=97ch code=40ch preview='def compute(context): | print("hello")'` while the file on disk now contains `print("hello1")`.

This is the known "edit-requires-iframe-reload" gap from v0.2.6
(`Forge: user vault mounted (N files; edits require iframe reload)`).
v0.2.17 closes it: after every disk write to a user-vault file,
sync the change into MEMFS AND invalidate the registry's cached
body for that snippet. Next compute sees fresh content.

### 2.1 Python: add `_forge_sync_user_file` helper

`src/pyodide-host.ts` inlined Python block — add near the other
`_forge_*` helpers (around line 432):

```python
def _forge_sync_user_file(relpath: str, new_body: str):
    """v0.2.17: write a single user-vault file to MEMFS and refresh
    the registry's cached entry for it. Called from JS after the
    plugin's writeGeneratedCode writes to disk via Obsidian's vault
    API. Keeps MEMFS in sync without re-scanning the whole user-
    vault mount — surgical, ~1ms per call.

    relpath is relative to the user-vault root (e.g. "Greet.md",
    "forge-moda/setup.md"). new_body is the full file content
    including frontmatter.
    """
    target = f"/bundle/user-vault/{relpath}"
    # Ensure parent dirs exist (writeGeneratedCode is called for
    # nested paths too).
    import os
    os.makedirs(os.path.dirname(target), exist_ok=True)
    with open(target, "w") as f:
        f.write(new_body)
    # Refresh the registry's cached entry. The exact API depends on
    # SnippetRegistry's surface — find it via grep against the
    # bundled engine source. Either:
    #   _forge_registry.refresh_file(relpath)  # if such a method exists
    # or
    #   _forge_registry.scan_file("/bundle/user-vault", relpath)
    # or
    #   _forge_registry._vaults["authoring"][snippet_id] = ...
    #     # (last resort — touches internals)
    # CC: investigate; pick the least-invasive option that works.
    _forge_registry.refresh_file(relpath)
```

### 2.2 JS: expose `syncUserVaultFile`

`src/pyodide-host.ts` — extend `PyodideHostInstance` interface and
impl:

```typescript
/** v0.2.17: sync a single user-vault file change into MEMFS so the
 *  next compute sees the new content. Call after every disk write
 *  to a user-vault file (writeGeneratedCode, manual editor saves
 *  caught via file-modify event, etc.). */
syncUserVaultFile(relPath: string, content: string): Promise<void>;

async syncUserVaultFile(relPath: string, content: string): Promise<void> {
  this.pyodide.globals.set('_forge_sync_relpath', relPath);
  this.pyodide.globals.set('_forge_sync_body', content);
  this.pyodide.runPython(`_forge_sync_user_file(_forge_sync_relpath, _forge_sync_body)`);
}
```

### 2.3 Wire from writeGeneratedCode

`src/main.ts:writeGeneratedCode` — after the `app.vault.modify`
call, sync MEMFS:

```typescript
for (const [id, code] of Object.entries(generated)) {
  // ... existing logic up to vault.modify ...
  const newContent = replacePythonSection(content, code);
  await this.app.vault.modify(file, newContent);

  // v0.2.17: keep MEMFS in sync. Otherwise Pyodide runs the
  // pre-modify body cached at plugin init.
  if (this._pyodideHost) {
    const host = await this._pyodideHost.getInstance();
    await host.syncUserVaultFile(file.path, newContent);
  }
}
```

`file.path` is Obsidian-vault-relative (no leading slash, no vault
name prefix). Matches the MEMFS layout that the user-vault mount
established.
