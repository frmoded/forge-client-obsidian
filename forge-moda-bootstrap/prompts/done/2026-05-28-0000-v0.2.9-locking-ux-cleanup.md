# v0.2.9 — Python locking UX revival + cleanup bundle

## Why this prompt exists

User reports: "There is no way to unlock the python facet now. Not
sure how the UI was in the past." The locking mechanism actually
**works**:

- Frontmatter `edit_mode: english | python` flips which facet is
  editable.
- `facet.ts` registers a CodeMirror transaction filter that blocks
  user-input transactions targeting the inactive facet
  (`facet.ts:210-234`).
- The toolbar shows a pencil/code icon button whose click handler
  is `toggleEditMode` (`main.ts:560-567`).
- Visual decoration ("read-only · switch to [X] mode to edit")
  marks the inactive facet (`facet.ts:116-145`).

The **discoverability** gap: that toolbar button is the **only**
unlock affordance. No Cmd+P entry, no right-click menu, no Notice
when entering Python mode explaining how to switch back. Easy to
miss; easy to assume "no way to unlock."

This prompt fixes discoverability without changing the underlying
mechanism. Plus three small cleanups: a dead settings field, a
vestigial parameter, and a server-side drift check.

Scope is deliberately small enough to drain in one CC pass; nothing
here changes wire formats, settings shapes that users depend on, or
the click-path behavior of the Forge button.

### 1.2 Right-click file-menu entry

```typescript

  this.app.workspace.on('file-menu', (menu, file) => {
    if (!(file instanceof TFile) || file.extension !== 'md') return;
    const fm = this.app.metadataCache.getFileCache(file)?.frontmatter;
    if (fm?.type !== 'action') return;
    const current = getEditMode(fm);
    const target = current === 'python' ? 'English' : 'Python';
    menu.addItem((item) => {
      item.setTitle(`Forge: Switch to ${target} editing mode`)
        .setIcon(current === 'python' ? 'pencil-line' : 'code')
        .onClick(async () => {
          // toggleEditMode reads the active view's file; we want
          // THIS file. Extract the body of toggleEditMode into a
          // helper that takes a TFile, then have both the toolbar
          // and this menu entry call it.
          await this.toggleEditModeForFile(file);
        });
    });
  }),
);
```

## 4. forge-transpile drift-check script (deferred bullet)

Add `forge-transpile/scripts/check-prompts-drift.sh` that compares
the prompts vendored in `forge-transpile/forge_transpile/anthropic_client.py`
(or wherever `build_user_prompt` lives — verify) against the
canonical source at `forge/forge/core/llm.py::_build_prompt`.

### 4.1 What "drift" means here

Both files build an Anthropic message body from a snippet inventory
(description + english + inputs + generation_notes + deps +
active_domains). The shape, ordering, formatting, and wording of
that message body is the contract between α and the engine — if
they diverge, α's `/generate` returns Python the engine wouldn't
have produced for the same input.

### 4.2 Script behavior

```bash
#!/usr/bin/env bash
# scripts/check-prompts-drift.sh
# Compares forge-transpile's vendored build_user_prompt against the
# engine's _build_prompt. Exits 0 if aligned, 1 if drift detected.

set -euo pipefail

ENGINE_SRC="${ENGINE_SRC:-../forge/forge/core/llm.py}"
LOCAL_SRC="forge_transpile/anthropic_client.py"

# Extract the relevant function bodies. The exact extraction is
# regex-based; if either file's structure changes, the script will
# need an update — flag this fragility in DEPLOY.md.

# 1. Extract _build_prompt from engine
engine_body=$(awk '/^def _build_prompt/,/^def [a-z]/' "$ENGINE_SRC" | head -n -1)

# 2. Extract build_user_prompt from local
local_body=$(awk '/^def build_user_prompt/,/^def [a-z]/' "$LOCAL_SRC" | head -n -1)

# 3. Compare normalized forms (strip whitespace + comments, sort
#    f-string field names alphabetically inside each block).
#    See diff output for guidance on which heuristic catches your
#    real drift cases.

if diff <(echo "$engine_body" | grep -v '^[[:space:]]*#' | tr -s '[:space:]') \
        <(echo "$local_body" | grep -v '^[[:space:]]*#' | tr -s '[:space:]'); then
  echo "PASS: prompts aligned"
  exit 0
else
  echo "FAIL: drift detected — see diff above"
  exit 1
fi
```

The actual diff heuristic is up to CC's judgment — the simpler the
better. If a strict line-by-line diff produces too many false
positives (because the two functions legitimately differ in argument
parsing while sharing prompt body), restrict the diff to the f-string
template literal that becomes the user-prompt body.
