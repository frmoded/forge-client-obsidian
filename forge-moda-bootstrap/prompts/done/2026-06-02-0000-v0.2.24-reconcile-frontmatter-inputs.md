# v0.2.24 — reconcile frontmatter `inputs:` with Python signature post-generate

## 1. Scope

After `/generate` writes a new Python facet, the plugin already
infers the input list from the Python signature at runtime
(v0.2.20 helper `_forge_get_input_names`). The modal opens
correctly with `name` field for Greet-shape snippets.

But the frontmatter `inputs:` field stays empty. A student reading
`Greet.md` sees:

```
---
type: action
description: Greet
inputs:           ← empty
---

# Python
def compute(context, name):    ← needs `name`
    print("hello " + name)
```

…and can't tell which is canonical. Architecturally Python wins
(option-b inference); for file-format readability, the two should
agree.

v0.2.24 closes the drift by **reconciling frontmatter `inputs:`
to match the inferred signature inputs**, immediately after
`writeGeneratedCode` lands the new Python.

What this prompt DOES:
- New helper that takes a `TFile` + the inferred-inputs list, writes
  them into the frontmatter via Obsidian's `processFrontMatter`.
- Wires the helper into `writeGeneratedCode` after the Python is
  written + MEMFS-synced (v0.2.17/.22), before the dependency-sync
  call.
- Updates the frontmatter only when it would change (idempotent — no
  churn on no-op writes).
- TDD failing-first test exercising the reconciliation path against
  production code via the v0.2.23 dynamic-load fixture pattern.

What this prompt does NOT:
- Touch α-side `/generate`. The hosted service stays as-is.
- Change `_forge_get_input_names` semantics. The helper is the source
  of truth; v0.2.24 just propagates its output to the file.
- Rename or restructure `lookupInventoryInputs`, `runSnippet`, or
  `writeGeneratedCode`'s broader shape. Surgical addition only.
- Address `generation_notes` or any other frontmatter field beyond
  `inputs`.

## 2. Why

User-side smoke against v0.2.23 confirmed the kwargs-drop bug is
closed and modal opens correctly. The remaining gap is the
frontmatter-vs-Python drift visible in the file itself — a
readability problem, not a functional one.

Closing it now (rather than deferring to v1.0) for two reasons:

1. Closed-beta students will read snippet files as documentation.
   A `Greet.md` with `inputs:` empty and `def compute(context,
   name)` invites the question "which is right?" that has no
   intuitive answer (Python via inference; obvious only after
   reading the architecture). Better to make the file
   self-consistent.

2. The fix is small (~30 lines) and shares all of v0.2.20-23's
   infrastructure: `_forge_get_input_names`, the dynamic-load
   fixture, the `_PYTHON_BLOCK_BEGIN/_END` markers. v1.0
   retrospective is heavier; this is a clean v0.2.x increment.

## 4. Implementation notes

### 4.1 Helper signature

```typescript
private async reconcileFrontmatterInputs(
  file: TFile,
  snippetId: string,
): Promise<void> {
  // Best-effort: if the host isn't wired or the helper throws,
  // log + skip. Frontmatter drift is a readability problem, not
  // a correctness one, so failing silently is the right
  // degradation.
  let inferred: string[];
  try {
    const hostManager = getPyodideHost();
    if (!hostManager) return;
    const host = await hostManager.getInstance();
    inferred = await host.getInputNames(snippetId);
  } catch (e) {
    console.warn(
      `Forge: getInputNames failed during frontmatter reconciliation for '${snippetId}'`,
      e,
    );
    return;
  }

  // Read current frontmatter to decide whether to write.
  const fm = this.app.metadataCache.getFileCache(file)?.frontmatter;
  const current: string[] = Array.isArray(fm?.inputs)
    ? (fm!.inputs as string[]).map(String)
    : [];

  // Idempotent check: no-op if the lists already match (order
  // matters — declared-first ordering is part of the contract).
  if (current.length === inferred.length
      && current.every((v, i) => v === inferred[i])) {
    return;
  }

  await this.app.fileManager.processFrontMatter(file, (fm: any) => {
    fm.inputs = inferred;
  });
}
```

### 4.2 Wiring into `writeGeneratedCode`

Add the call AFTER `host.syncUserVaultFile(...)` (which already
runs per v0.2.17) and BEFORE `syncDependencies(...)`:

```typescript
// Inside the for-loop over generated entries:
const content = await this.app.vault.read(file);
const newContent = replacePythonSection(content, code);
await this.app.vault.modify(file, newContent);

try {
  const hostManager = getPyodideHost();
  if (hostManager) {
    const host = await hostManager.getInstance();
    await host.syncUserVaultFile(file.path, newContent);
  }
} catch (e) {
  console.warn(`Forge: MEMFS sync after write failed for '${id}'`, e);
}

// v0.2.24: reconcile frontmatter inputs with the just-written
// Python signature. Best-effort; failure is non-fatal (drift
// is a readability concern, not a correctness one).
await this.reconcileFrontmatterInputs(file, id);

// (v0.2.17 sync_dependencies call stays as-is below)
```

### 4.3 Ordering rationale

- `syncUserVaultFile` BEFORE reconciliation ensures
  `_forge_get_input_names` sees the just-written Python.
- Reconciliation BEFORE `syncDependencies` because dependencies
  are derived from the Python body (not the frontmatter); putting
  them after gives the cleanest sequence: Python written → MEMFS
  fresh → frontmatter agrees → dependency sync runs (or fails
  gracefully if no engine).

### 4.4 `processFrontMatter` semantics

Obsidian's `app.fileManager.processFrontMatter(file, fn)` mutates
the file's frontmatter via a callback that receives the parsed
object. It writes back via the same yaml serializer Obsidian uses
for in-place edits. Two important details:

- Setting `fm.inputs = ['name']` produces YAML `inputs:\n  - name`
  (multi-line list form). Compatible with Forge's existing
  frontmatter readers.
- Empty list (`fm.inputs = []`) produces `inputs: []` (flow-style).
  Also compatible. The reconciliation should NEVER produce `inputs:`
  with no value — either the list is non-empty (specific params)
  or the key gets deleted (`delete fm.inputs`).

For now: just set `fm.inputs = inferred` even when empty. The
flow-style `[]` is fine. Document in §5 if smoke surfaces a
readability concern.

### 4.5 Edge cases

- **File deleted between `vault.modify` and reconciliation**:
  `processFrontMatter` throws. Caught by the try/catch around
  the helper at the call site; logs + continues.
- **Snippet body has no `compute()` function**: `getInputNames`
  returns the frontmatter `declared` value as fallback. If the
  user's frontmatter was empty, this is `[]` — reconciliation
  no-ops (already empty).
- **Declared frontmatter inputs have extras beyond Python sig**:
  `getInputNames` returns declared-first + sig-only-appended. The
  union preserves user intent. If user explicitly removed a param
  from their Python signature but left it in frontmatter, the
  reconciliation will preserve it — that's a feature, not a bug
  (user intent wins over inference for declared params).

## 5. Tests

### 5.1 TDD discipline (HARD RULE per cc-prompt-queue.md §57-78)

Failing test FIRST. Per §80, the test must exercise PRODUCTION code
via dynamic load — the v0.2.23 fixture pattern in
`src/compute-kwargs.test.ts:bootGreet` already does this for the
inline Python block, so the new test reuses it.

The reconciliation is on the JS-side (TypeScript), not in Pyodide.
The test can't dynamically load TypeScript at runtime (no
equivalent to the Python `runPython` trick). Two options:

**Option A (preferred): extract `reconcileFrontmatterInputs`'s
core logic into a pure-core helper.** Move the
"compare-then-decide-to-write" logic into
`src/frontmatter-inputs-reconcile.ts`:

```typescript
export interface InputsReconcileAdapter {
  getInferredInputs(snippetId: string): Promise<string[]>;
  readCurrentInputs(): string[];
  writeInputs(next: string[]): Promise<void>;
}

export async function reconcileInputs(
  snippetId: string,
  adapter: InputsReconcileAdapter,
): Promise<{ status: 'wrote' | 'no-op' | 'skipped'; inputs: string[] }> {
  let inferred: string[];
  try {
    inferred = await adapter.getInferredInputs(snippetId);
  } catch {
    return { status: 'skipped', inputs: [] };
  }
  const current = adapter.readCurrentInputs();
  if (current.length === inferred.length
      && current.every((v, i) => v === inferred[i])) {
    return { status: 'no-op', inputs: current };
  }
  await adapter.writeInputs(inferred);
  return { status: 'wrote', inputs: inferred };
}
```

Then `main.ts`'s `reconcileFrontmatterInputs` is the
Obsidian-coupled glue:

```typescript
private async reconcileFrontmatterInputs(file: TFile, snippetId: string): Promise<void> {
  await reconcileInputs(snippetId, {
    getInferredInputs: async (id) => {
      const hostManager = getPyodideHost();
      if (!hostManager) throw new Error('Pyodide host not wired');
      const host = await hostManager.getInstance();
      return host.getInputNames(id);
    },
    readCurrentInputs: () => {
      const fm = this.app.metadataCache.getFileCache(file)?.frontmatter;
      return Array.isArray(fm?.inputs)
        ? (fm!.inputs as unknown[]).map(String)
        : [];
    },
    writeInputs: async (next) => {
      await this.app.fileManager.processFrontMatter(file, (fm: any) => {
        fm.inputs = next;
      });
    },
  });
}
```

Pure-core tests in `src/frontmatter-inputs-reconcile.test.ts` use
an in-memory `InputsReconcileAdapter` stub. No obsidian shim
needed.

**Option B**: Skip the extraction. Test only via integration —
mount the engine, call the reconciliation through a TFile stub.
More invasive, requires a TFile shim.

CC: pick A (extraction). It's the established pattern (closed-beta-ux.ts,
copy-dir-core.ts, forge-toml-stub.ts, etc. — 8th-going-on-9th
extraction).

- **Don't ship a fix without the failing test landing first.**
  Per cc-prompt-queue.md hard rule.
- **Don't touch α-side `/generate`.** Scope is plugin-side
  reconciliation only.
- **Don't reuse the v0.2.23 `_PYTHON_BLOCK_BEGIN`/`_END` extraction**
  for this prompt's tests. The reconciliation is JS-side; pure-core
  extraction is the right pattern.
- **Don't add a 6th case or expand scope.** The 5 cases cover the
  contract. If smoke surfaces another shape, that's a separate
  prompt.
- **Don't change the post-write call ordering** (sync MEMFS →
  reconcile frontmatter → sync deps). The order is load-bearing
  per §4.3.
- **Don't make the reconciliation fatal.** Wrap in try/catch at
  the call site; log + continue. Frontmatter drift is a
  readability concern, not a correctness one — a failing
  reconciliation must not block compute.
