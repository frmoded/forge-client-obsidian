# Engine-bundle drift-check — sync script + release-zip preflight

## Why this prompt exists

forge-music's v0.2.28 feedback flagged the engine bundle drift gap
as their strongest cross-cutting concern. Concrete pain: fifth
manual `cp -r ~/projects/forge/forge/{core,moda,music,__init__.py}
~/projects/forge-client-obsidian/assets/engine/forge/` cycle in
three days; one real gap was caught accidentally during v0.2.27 +
v0.2.28 work (the `forge.music.lib` module was silently missing
from the bundle until v0.2.27 explicitly copied it). The drift is
operational toil and a real correctness risk: any forge-engine
edit needs a matching plugin-bundle copy, or the engine layer
diverges between the source-of-truth and the shipped artifact.

This prompt closes the gap with two artifacts:

1. **`npm run sync-engine-bundle`** — idempotent copy script.
2. **Release-zip preflight assertion** — fails the release-zip
   build if drift exists.

Together: the script keeps the bundle current during development;
the preflight forces the script-run as part of release discipline.

This is forge-core cross-cutting work (touches engine repo
boundaries + plugin build pipeline + release process). Not a v1.0
audit defer.

## 1. Scope

**Source of truth**: `~/projects/forge/forge/` (the forge engine
repo, on disk).

**Mirror target**: `~/projects/forge-client-obsidian/assets/engine/forge/`.

**Files in scope** (everything Pyodide imports at runtime; identify
empirically by reading `pyodide-host.ts` inline Python's `import`
lines):

- `forge/__init__.py` (top-level package marker)
- `forge/core/` (recursively — all `.py` files; the engine kernel)
- `forge/moda/` (recursively — moda domain)
- `forge/music/` (recursively — music domain)

**Excluded** (engine source has more than the plugin needs; bundle
is intentionally subset):

- `forge/api/` — uvicorn HTTP server; not used in V1's Pyodide path.
- `forge/installer/` — pre-V1 deploy machinery.
- `forge/sdk/` — Python client for HTTP backend; not needed.
- `forge/builtins/` — install / registry snippets; deliberately not
  bundled (per V1's "bundled-only, no install" decision; see
  Constitution A5.3).
- `forge/config.py` — engine config; the bundle uses inline values.
- `forge/__pycache__/` — generated bytecode; never bundled.
- `tests/` — test suite; not bundled.

The script and preflight both apply the same scope filter; drift
is detected only on in-scope files.

## 3. Fix shape

### 3.1 Pure-core helper

`src/engine-bundle-drift-core.ts` (new file, pure-core per §80
extraction convention). Exports:

```typescript
export interface DriftEntry {
  relPath: string;  // relative to engine forge/ root, e.g. 'core/registry.py'

export interface BundleDriftAdapter {
  listEngineFiles(scope: 'source' | 'bundle'): Promise<string[]>;  // returns scope-filtered rel paths
  readFile(scope: 'source' | 'bundle', relPath: string): Promise<Buffer>;
}

export async function engineBundleDrift(
  adapter: BundleDriftAdapter,

  // Walk both, compare, produce drift list.
}
```

10th pure-core extraction in the arc. Structural adapter type
keeps the helper testable without filesystem coupling.

### 3.2 `npm run sync-engine-bundle` script

`scripts/sync-engine-bundle.mjs` (new). Uses the same scope filter
as the drift helper. Steps:

1. Resolve source path: `${process.cwd()}/../forge/forge/`. Fail
   loudly if not found (with a clear "is the forge repo cloned
   at sibling?" message).
2. For each in-scope file in source, copy to corresponding bundle
   path. Use `fs.copyFileSync` (atomic for single files; the
   bundle dir survives mid-script crash).
3. For each in-scope file in bundle NOT in source, delete from
   bundle (cleans up `'orphaned-in-bundle'` drift).
4. Log every action: `[copy] forge/core/registry.py`, `[delete-orphan] forge/core/old_helper.py`.
5. Final summary: `Synced N files, M deletions.`

Add to `package.json`:

```json
"scripts": {
  "sync-engine-bundle": "node scripts/sync-engine-bundle.mjs"
}
```

### 3.3 Preflight assertion in `build-release-zip.mjs`

Before the zip-build step, run `engineBundleDrift` against the
filesystem adapter. If drift exists, fail loudly with the drift
list printed and a one-line hint: `Run 'npm run sync-engine-bundle' to resolve.`

The preflight is **build-time gating**: developers can have local
drift during work (the script lets them sync), but no release zip
ships with engine-bundle drift.

### 3.4 README note

Append to `forge-client-obsidian/README.md` a small "Engine bundle
sync" section: when modifying `~/projects/forge/forge/core/` (or
moda/, or music/), run `npm run sync-engine-bundle` before building
the plugin. Explain that the preflight catches drift but the script
is the workflow shortcut.

## 5. Manual smoke

After CC ships:

1. **Drift detection in working state**: with a clean tree, run
   `npm run build`. Expect: preflight reports no drift; build
   succeeds.
2. **Drift detection on real drift**: edit
   `~/projects/forge/forge/core/registry.py` (add a comment line).
   Run `npm run build`. Expect: preflight FAILS with
   `core/registry.py: content-mismatch`. The drift output names
   the file.
3. **Sync resolves drift**: run `npm run sync-engine-bundle`.
   Expect: `[copy] forge/core/registry.py` logged. Then re-run
   `npm run build`. Expect: passes.
4. **Orphan detection**: temporarily delete a file from source
   (`mv ~/projects/forge/forge/core/snippet_registry.py /tmp/`).
   Run `npm run build`. Expect: preflight fails with
   `core/snippet_registry.py: orphaned-in-bundle`. Restore
   the file (`mv /tmp/snippet_registry.py
   ~/projects/forge/forge/core/`). Re-run sync. Build passes.

## 6. Version bump + release

Tactical operational change; no plugin version bump unless the
fix accidentally changes plugin runtime behavior (unlikely for a
sync script + preflight). If CC's audit surfaces a real runtime
fix as part of the sync (e.g., a file the plugin needed that
wasn't in the bundle until now), bump to v0.2.29; otherwise skip
the version bump and just commit the tooling.
