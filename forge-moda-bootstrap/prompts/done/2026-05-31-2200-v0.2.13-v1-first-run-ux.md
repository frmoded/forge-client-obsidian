# v0.2.13 — V1-accurate first-run UX (rewrite welcome, extract forge-moda, hide install menu)

## Why this prompt exists

A fresh-vault install via `forge-installer` v0.1.1 surfaces three
connected V1 gaps:

1. **`Welcome.md` (auto-created in fresh vaults) is pre-V1.** The
   `WELCOME_NOTE` in `src/welcome.ts` tells the user to `Zap
   [[install]] "forge-core"` — references a vault that isn't
   bundled and an install flow whose snippets aren't bundled either.
   Students read it as a tutorial; it behaves as a trap.
2. **The bundled engine excludes `forge/builtins/`.** Only
   `forge/core/` and `forge/moda/` got copied into
   `assets/engine/`. The `install` snippet at
   `forge/builtins/snippets/install.md` is missing, along with all
   its dependencies (`registry/lookup`, `registry/fetch`,
   `vault/extract`, `manifest/add_dep`). Even if bundled, V1 has
   no hosted registry to query — the flow can't succeed.
3. **The ribbon Forge → "Update installed domain vaults"** menu
   item triggers the broken install flow. User-visible
   `SnippetResolutionError: Snippet 'install' not found`.

V1 design (per v1-deployment-plan) is "bundled forge-moda runs from
plugin assets regardless of vault content" — no install needed. The
UI just hasn't caught up. This prompt aligns the UI with the design.

Out of scope (deferred to v1.1+): bundling `forge/builtins/` and
standing up a registry. V1 simply doesn't need them.

## 2. First-run extracts bundled forge-moda into vault root

`src/welcome.ts` — extend `runFirstRunCheck` to copy the bundled
forge-moda library into the user's vault root, alongside the
Welcome.md write.

Bundled location:
`.obsidian/plugins/forge-client-obsidian/assets/vaults/forge-moda/`

Target location: `forge-moda/` (vault root).

Behavior:
- Only on first-run (gated by the existing sentinel check; doesn't
  fire for existing-install vaults).
- Plus: independent check for "forge-moda/ exists in vault" — if
  missing on any plugin load (not just first-run), extract anyway.
  This covers the upgrade-without-content case where sentinel is
  true but the vault has no forge-moda content.
- Skip if `<vault>/forge-moda/` already exists (don't overwrite
  user-modified content).

```typescript
export async function runFirstRunCheck(app: App): Promise<void> {
  // ... existing sentinel + Welcome.md logic ...

  // v0.2.13: extract bundled forge-moda if the vault doesn't have it.
  // Independent of sentinel — covers both fresh vaults (sentinel
  // absent) and upgrade-without-content vaults (sentinel present
  // but no forge-moda dir yet).
  await ensureBundledForgeModa(app);
}

async function ensureBundledForgeModa(app: App): Promise<void> {
  const adapter = app.vault.adapter;
  const targetDir = 'forge-moda';

  if (await adapter.exists(targetDir)) {
    return;  // user-modified or previously-extracted; leave alone
  }

  const sourceDir = '.obsidian/plugins/forge-client-obsidian/assets/vaults/forge-moda';
  if (!(await adapter.exists(sourceDir))) {
    console.warn('Forge: bundled forge-moda missing from plugin assets; skipping extraction');
    return;
  }

  await copyDirRecursive(adapter, sourceDir, targetDir);
  console.log('Forge: extracted bundled forge-moda into vault');
}

async function copyDirRecursive(
  adapter: DataAdapter,
  src: string,
  dst: string,
): Promise<void> {
  await adapter.mkdir(dst);
  const listing = await adapter.list(src);
  for (const filePath of listing.files) {
    const name = filePath.slice(src.length + 1);
    const data = await adapter.readBinary(filePath);
    await adapter.writeBinary(`${dst}/${name}`, data);
  }
  for (const dirPath of listing.folders) {
    const name = dirPath.slice(src.length + 1);
    await copyDirRecursive(adapter, dirPath, `${dst}/${name}`);
  }
}
```

Adapter API notes:
- `app.vault.adapter.list(path)` returns `{files, folders}` — both
  full paths from vault root.
- `readBinary` / `writeBinary` for arbitrary file content (handles
  text and binary alike).
- `mkdir` is non-recursive on some platforms — `copyDirRecursive`
  walks manually (same pattern as `forge-installer`'s
  `ensureParentDir`).

Error handling: log + continue. A missing bundle shouldn't crash
plugin load; the simulator and own-snippet authoring still work
without forge-moda extracted.

## 3. Hide "Update installed domain vaults" menu

`src/forge-action.ts:143` — the menu item is registered as one of
several actions when the user clicks the ribbon Forge icon. Two
options:

**Option A (recommended): remove the menu item entirely.**

Delete lines 143-144:
```typescript
.addItem(i => i.setTitle('Update installed domain vaults').setIcon('refresh-cw')
  .onClick(() => updateDeclaredVaults(host, declared)));
```

The supporting functions (`updateDeclaredVaults`, `installVault`)
become dead code. Leave them in place for now — same audit-cleanup
pass at v1.0 that handles the other tactical dead code (task #19
in v1-deployment-plan). Removing live code paths is the priority;
dead helpers are inert.

**Option B (alternative): no-op the click with a Notice.**

Replace the onClick handler with:
```typescript
.onClick(() => new Notice(
  'V1 closed-beta has no remote vault registry. Bundled forge-moda '
  + 'runs from plugin assets; no install needed.'
));
```

This keeps the menu item visible but informative.

Pick A for the cleanest closed-beta UX. The menu item itself is
confusing for students who don't have "declared domain vaults" —
they shouldn't see it at all.

Two other call sites of `installVault` at `forge-action.ts:451`
and `:805` — check whether those are also user-reachable in V1.
If yes, apply the same removal pattern. If no (e.g., they're
called from a "first-run install" path that the new
`ensureBundledForgeModa` replaces), they're dead code too.
Document findings in feedback §3.
