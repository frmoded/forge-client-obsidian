## Implementation notes

### Cross-references between blues snippets

If the blues set has internal callees (e.g., `chorus` calls `twelve_bar_blues_progression`, `song` calls `chorus`, `form` calls `twelve_bar_blues_progression`), make sure ALL of them are migrated together so the dep graph is intact in the published vault.

If a blues snippet calls something OUTSIDE the blues set (e.g., a helper from `forge.music.lib`), that's fine — `forge.music.lib` is engine-side and already shipped.

### Pre-injected music21 names

The engine's executor pre-injects music21 module aliases (`music21`, `stream`, `note`, `chord`, etc. per `executor.py:14-30`). Blues snippets that use those names without explicit `import music21` are correct — the engine handles it. Don't add `import music21` to migrated snippets unless they were already importing it.

### Test fixture vault path

The test fixture needs to point at the migrated forge-music vault. Use `~/projects/forge-music/` directly — there's no `forge-music-vault` authoring layer today. If the test infrastructure expects a specific layout (e.g., snippets under a `forge-music/` subdir), adjust the fixture accordingly.

### Publish gotchas

The publish script (`publish-vault.sh`) requires the forge-music repo to have a clean working tree and no unpushed commits. Per the 0200 fix, the author's pre-bump is honored. CC commits the content + bump first, then runs the publish script which adds the tag.

If forge-music isn't currently a git repo or doesn't have a clean upstream, flag in the report and don't try to force-push.
