## 2. Fix shape

### 2.1 Register `vault.on('modify')` in plugin onload

`src/main.ts` — in `onload()`, alongside the other workspace event
registrations (after the v0.2.13 file-menu hook from v0.2.9 if
that's where similar hooks live; otherwise pick a sensible spot
near workspace setup):

```typescript
this.registerEvent(
  this.app.vault.on('modify', async (file) => {
    if (!(file instanceof TFile)) return;
    if (file.extension !== 'md') return;
    if (this.shouldSkipForMemfsSync(file.path)) return;

    try {
      const hostManager = getPyodideHost();
      if (!hostManager) return;
      const host = await hostManager.getInstance();
      const content = await this.app.vault.read(file);
      await host.syncUserVaultFile(file.path, content);
    } catch (e) {
      console.warn(`Forge: MEMFS sync on modify failed for '${file.path}'`, e);
    }
  }),
);
```

Wrap the body in a try/catch (non-fatal — same pattern as the
post-writeGeneratedCode sync in v0.2.17). Console-warn on failure;
the user's edit isn't lost on disk, just the MEMFS update fails.

### 2.2 Path-skip predicate

Add `shouldSkipForMemfsSync(path: string): boolean` as a private
method on the plugin class (or pure-core helper if it grows beyond
a few cases). Skip:

- Anything under `.obsidian/` — plugin data, workspace state, etc.
- Anything under `.forge/` — sentinel + future cache files.
- Anything under `.trash/` — deleted notes.
- Any path that doesn't end in `.md` (already filtered above, but
  belt-and-suspenders).

```typescript
private shouldSkipForMemfsSync(path: string): boolean {
  if (path.startsWith('.obsidian/')) return true;
  if (path.startsWith('.forge/')) return true;
  if (path.startsWith('.trash/')) return true;
  return false;
}
```

The bundled assets folder (`assets/vaults/forge-moda/`,
`assets/vaults/forge-music/`) lives under `.obsidian/plugins/...` so
the `.obsidian/` skip covers it — users editing files inside the
plugin install dir is not a vault-side operation.

### 2.3 Obsidian's autosave throttle is the debounce

`vault.on('modify')` fires on Obsidian's autosave (typically ~1
second after the user stops typing) AND on explicit Cmd+S. **No
extra debounce needed** — Obsidian throttles for us. Burst-write
scenarios (e.g., a script writing 100 files in 100ms) would benefit
from app-side debouncing, but interactive editing doesn't.

If the auto-smoke surfaces flakiness from rapid fire (e.g., the
test suite's own writes triggering modify-loops), add a debounce
THEN — not preemptively. Flag in feedback §7 if added.

### 2.4 Keep the explicit sync inside writeGeneratedCode

The v0.2.17 `await host.syncUserVaultFile(file.path, newContent)`
inside `writeGeneratedCode` stays. The two sync points cover
different timing:

- writeGeneratedCode's explicit sync: immediate, completes BEFORE
  the next compute call. Critical for the /generate → write →
  compute sequence to see fresh state on the same click.
- vault.on('modify') hook: covers direct editor edits made between
  Forge-clicks. Async, fires after Obsidian's autosave.

Both fire on the same writeGeneratedCode-driven edit. That's
redundant but harmless — both sync the same content. The
modify-event fire happens AFTER compute (~1s autosave delay), so
it doesn't compete with the explicit sync for the immediate next
compute.
