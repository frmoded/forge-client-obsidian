# v0.2.21 — pre-flight sync before getInputNames (close runSnippet race)

## Why this prompt exists

v0.2.20 added `_forge_get_input_names(snippet_id)` + JS-side wiring
in `runSnippet` to ask the engine for the Python-signature-augmented
input list. The 9 test cases in `src/input-inference.test.ts`
all pass.

User-side smoke STILL fires the original `TypeError: compute()
missing 1 required positional argument: 'name'` after upgrading
to v0.2.20. Diagnostic trace:

```
Forge: stock packages loaded in 1284ms
Forge: user vault mounted (1 files; edits require iframe reload).
Forge: bundle mounted in 1312ms
Forge: engine ready in 1785ms
Forge Compute → {snippetId: 'Greet', args: Array(0), inputs: {…}}
Forge debug: run_snippet('Greet') body=154ch code=54ch preview='def compute(context, name): | print("hello " + name)'
Forge Pyodide compute failed: PythonError: ... TypeError: compute() missing 1 required positional argument: 'name'
```

Two things to notice:

1. **No `Forge: generate (α)` line.** `forgeSnippet` (`main.ts:1185`)
   went through the `getEditMode(fm) === 'python'` branch on line
   1192 → skipped `/generate` entirely → went straight to
   `runSnippet`. So v0.2.19's pre-flight sync (which lives inside
   `generate()`) never fired.

2. **The diagnostic `body=154ch preview='def compute(context, name)'`
   IS the fresh body** — but that prints from inside
   `_forge_run_snippet` AFTER the modal already submitted with
   empty kwargs. By then it's too late.

The race: `runSnippet` calls `host.getInputNames(snippetId)` →
helper reads from `_forge_resolver.resolve(snippet_id)` →
registry returns the CACHED entry. If the cached body is stale
(modify-hook hasn't fired or completed), the helper parses the
OLD Python signature, finds no `name`, returns `[]`. Modal opens
with zero fields → submit → compute → TypeError.

CC's v0.2.20 tests pass because each test reboots the registry
with a fresh body before calling the helper. Production registry
state is sticky between scans; refresh_file is the only way to
update an entry's cached body without a full re-scan.

This prompt closes the race by mirroring v0.2.19's preflight
pattern: `getInputNames` should also refresh the registry from
current MEMFS state before parsing the signature.

## 2. Fix shape — option (b): bake refresh into the helper

The helper itself should refresh the registry from MEMFS before
parsing. This makes it race-free from any caller's perspective —
JS-side doesn't need to remember to pre-flight.

`src/pyodide-host.ts` inlined Python — modify
`_forge_get_input_names`:

```python
def _forge_get_input_names(snippet_id: str):
    """v0.2.21: derive input names from Python signature + frontmatter.

    Race-proofed: refreshes the registry's cached entry from current
    MEMFS state before parsing, so callers don't need to pre-flight.
    Mirrors v0.2.19's preflightThenInventory pattern.
    """
    # v0.2.21 race fix: refresh registry from MEMFS first. The
    # registry caches snippet bodies at scan-time; if the user
    # edited the file after plugin load and the modify-hook hasn't
    # fired yet, resolve() returns stale. Refreshing here makes the
    # helper race-free from runSnippet's call site.
    relpath = f"/bundle/user-vault/{snippet_id}.md"
    try:
        _forge_registry.refresh_file(relpath)
    except Exception:
        # Path may not exist (bundle-only snippet, etc.). Let
        # resolve() raise its own error.
        pass

    # Rest of the existing v0.2.20 logic — unchanged.
    import ast as _forge_ast
    snip = _forge_resolver.resolve(snippet_id)
    meta = snip.get("meta") or {}
    declared = [str(i) for i in (meta.get("inputs") or [])]

    body = snip.get("body") or ""
    try:
        code = extract_python(body)
    except Exception:
        return declared
    if not code or not code.strip():
        return declared
    # ... rest of v0.2.20 helper unchanged
```

**No JS-side change** needed for the production fix. The helper
becomes self-sufficient.

If CC's investigation finds that the JS-side wiring in
`runSnippet` ALSO benefits from an explicit `syncUserVaultFile`
before `getInputNames` (for the additional safety of reading
fresh DISK content via `app.vault.read` and pushing it to MEMFS),
add that too — same pattern as v0.2.19's pre-flight in
`generate()`. Document in feedback §6 if added.

Both layers are belt-and-suspenders: helper-side refresh + JS-
side disk-to-MEMFS sync = no race regardless of timing.

## 6. One observation worth flagging

This is the third "production race that suite tests miss" pattern:
- v0.2.18 → vault.on('modify') async vs synchronous Forge-click
- v0.2.19 → generate() needed sync before /generate
- v0.2.21 → runSnippet's getInputNames also needed sync before resolve

The systemic issue: **any function that reads from the registry's
cached entry has a stale-data risk window unless it refreshes
first**. The pattern is now clear: the engine helper should
refresh-self before resolving, not rely on JS-side pre-flight.
Worth noting in feedback §6 — once this lands, audit all other
registry-resolve sites for the same pattern. Candidate sites:
`_forge_get_generate_inventory` (v0.2.19 already wrapped it via
preflightThenInventory), `_forge_list_snippets`, anywhere else
that calls `_forge_resolver.resolve(...)`.
