# v0.2.7 — console polish for closed-beta: silence dep-sync ECONNREFUSED, add first-run welcome

## Why this prompt exists

v0.2.6 cleared the closed-beta blocker (connectVault no longer
needs uvicorn), but two ergonomic issues remain that students will
hit on first install:

1. **Scary yellow warning on every Forge-click.** With no uvicorn
   running, `writeGeneratedCode` in main.ts calls
   `syncDependencies(localhost:8000, ...)` post-`/generate` and the
   network refusal surfaces as:
   ```
   Forge: sync_dependencies failed for 'simulation'
   Error: net::ERR_CONNECTION_REFUSED ...
   ```
   Functionally harmless (the Python facet was already written; the
   call stack confirms control returns to `writeGeneratedCode` and
   compute proceeds), but a student opening dev console for any
   other reason will see this red-edged stack and worry.

2. **Silent first-run UX.** Plugin installs, student hasn't pasted
   their transpile token yet, Forge-click on a snippet without a
   Python facet → they see the v0.2.4 "Set your transpile token…"
   Notice **only when they try to use it**. A welcome notice on
   first plugin load would route them to settings before they hit
   the error state.

Out of scope for v0.2.7: routing /sync_dependencies through Pyodide
(v1.0+), routing /canonicalize through α (v1.1), removing the dead
HTTP fallbacks (v1.0+).

## 1. Silence the sync_dependencies warn on network refusal

`forge-client-obsidian/src/main.ts:1245-1251`, current:

```typescript
try {
  await syncDependencies(this.settings.serverUrl, vaultPath, id);
} catch (e) {
  console.warn(`Forge: sync_dependencies failed for '${id}'`, e);
}
```

The plain `console.warn` fires on every Forge-click in closed-beta
config. Replace with a session-scoped guard that downgrades network-
refusal errors to a single info-level message:

```typescript
try {
  await syncDependencies(this.settings.serverUrl, vaultPath, id);
} catch (e) {
  const msg = e instanceof Error ? e.message : String(e);
  const isNetRefusal = /ERR_CONNECTION_REFUSED|ECONNREFUSED|ENOTFOUND|ENETUNREACH/i.test(msg);
  if (isNetRefusal) {
    if (!this._b7SyncSkippedLogged) {
      console.info(
        'Forge: dependency sync skipped — no local engine reachable at ' +
        `${this.settings.serverUrl}. This is expected in closed-beta; ` +
        'dependency-section refresh is engine-side only.'
      );
      this._b7SyncSkippedLogged = true;
    }
  } else {
    // Non-network failure — keep the warn so real bugs surface.
    console.warn(`Forge: sync_dependencies failed for '${id}'`, e);
  }
}
```

Add the field next to the other private state declarations in the
plugin class:

```typescript
// v0.2.7: tracks whether we've logged the "no local engine" info
// once already this session. Avoids spamming the console on every
// Forge-click in closed-beta where uvicorn is intentionally absent.
private _b7SyncSkippedLogged = false;
```

Reset to `false` in `onunload` so a Cmd+R reload re-fires the one-
time message (useful for debugging without restarting Obsidian).

**Do not** change the matching `syncEdgesForActive` path
(main.ts:1255-) — that's an explicit user-invoked "Sync edges"
command, which SHOULD loudly fail if the engine isn't reachable
(the user expressly asked for the sync). Leave its error handling
as-is.

## 2. First-run welcome notice

Add to the plugin class state:

```typescript
// v0.2.7: tracks whether the welcome notice has been shown.
// Persisted via the same data.json that holds settings — once true,
// stays true across reloads. Re-fires for a true first install only.
private seenWelcome = false;
```

Persist via `loadData()` / `saveData()`. Extend `ForgeSettings` or add
a sibling persisted-state object — choose whichever pattern fits the
existing code (don't break existing settings shape).

In `onload` (after settings load, after Pyodide init, but before any
user interaction can happen), add:

```typescript
if (!this.seenWelcome) {
  const hasToken = !!this.settings.transpileServiceToken?.trim();
  const noticeText = hasToken
    ? 'Forge is ready. Open Settings → Forge to review your configuration.'
    : 'Welcome to Forge. To enable AI-powered snippet generation, paste your transpile token at Settings → Forge → Transpile service → Transpile service token.';
  new Notice(noticeText, 10000);  // 10-second display
  this.seenWelcome = true;
  await this.saveData(/* updated state */);
}
```

The branched message handles both first-install paths: token-already-
set (rare, but possible if data.json was migrated) gets the shorter
ack; the no-token case (closed-beta-typical) gets the explicit setup
nudge.

10-second Notice (instead of the default 5) because closed-beta
students may be mid-other-task when Obsidian finishes loading; 5
seconds is too easy to miss.

## 6. Manual smoke guidance for user

1. **Install v0.2.7** per INSTALL.md (drag-and-drop replacing the
   existing v0.2.6 folder).
2. **First-run welcome.** Before opening Settings, observe the
   Notice on plugin load:
   - If `seenWelcome` was already true from v0.2.6 data.json (likely
     because the field didn't exist before, defaulted to false on
     first v0.2.7 load) — Notice fires once now, then never again.
   - Confirm the message matches the token-status branch (with-token
     vs no-token version).
3. **Quiet console on Forge-click.** With uvicorn off and serverUrl
   at `http://localhost:8000`:
   - Clear dev console.
   - Forge-click a snippet whose Python facet you've deleted.
   - Expect: success flow as in v0.2.6, but the
     `sync_dependencies failed` warn is **gone** on the first click
     and an `info`-level "dependency sync skipped" line appears once.
   - Forge-click a second snippet. Expect: no additional dep-sync
     log line (the once-per-session guard held).
4. **Real failure still surfaces.** This requires injecting a non-
   network error, which is awkward to do manually — skip this check
   in manual smoke; covered by §4 unit test.
