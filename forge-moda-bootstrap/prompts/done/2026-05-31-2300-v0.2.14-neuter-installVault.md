# v0.2.14 — neuter installVault for V1 (no registry, nothing to install)

## Why this prompt exists

v0.2.13 removed the "Update installed domain vaults" ribbon menu
item — one of three `installVault` call sites that triggered a
`SnippetResolutionError: Snippet 'install' not found`. CC's audit
flagged the other two; today's smoke confirms they're equally
reachable in V1:

> User tried to "add music settings" (likely via the EditVaultDomainsModal
> at `forge-action.ts:453`, OR the InitializeForgeVaultWizard at
> `:807` — either way `installVault` ran). Same SnippetResolutionError.

V1 design (per v1-deployment-plan): forge-moda is bundled, no hosted
registry, no install needed. `installVault` is dead weight that
shouldn't get invoked at all. This prompt neuters it at the source
plus prevents the two known auto-open paths.

## 1. Replace `installVault` body with a V1-aware Notice

`src/forge-action.ts:235` `installVault` — replace the entire body
with a graceful "V1 doesn't install" path:

```typescript
async function installVault(host: ForgeHost, vaultName: string): Promise<boolean> {
  // v0.2.14: V1 closed-beta has no hosted vault registry; the engine
  // `install` snippet that this called isn't bundled either. Three
  // call sites still reach here (forge-action.ts:168, 453, 807); rather
  // than ripping out each call site, neuter the function and surface
  // a clear Notice. Cleanup of the dead call paths is on the v1.0
  // audit (task #19 in v1-deployment-plan).
  console.warn(`Forge: install requested for "${vaultName}" — V1 closed beta does not support remote vault install.`);
  new Notice(
    `Forge: install of "${vaultName}" skipped — V1 closed beta has `
    + 'no remote vault registry. forge-moda is pre-bundled; '
    + 'additional vaults (e.g. music) are deferred to v1.1+.',
    10000,  // 10s — explanation needs reading time
  );
  return false;
}
```

The boolean return matches the existing call-site contract — every
caller already branches on the boolean and surfaces a separate
failure notice when `false`. The internal Notice here is just the
authoritative explanation; the call-site notices remain as
secondary "couldn't add this dep" surfaces.

## 2. Pre-populate `forge.toml` at first-run

`src/welcome.ts` — extend `runFirstRunCheck` (after the existing
`ensureBundledForgeModa` call) with `ensureForgeToml(app)`:

```typescript
async function ensureForgeToml(app: App): Promise<void> {
  const adapter = app.vault.adapter;
  const tomlPath = 'forge.toml';

  if (await adapter.exists(tomlPath)) {
    return;  // user-managed or previously-created; leave alone
  }

  // v0.2.14: a minimal forge.toml at the vault root stops the
  // InitializeForgeVaultWizard from auto-opening on first ribbon
  // Forge click. The wizard's auto-open trigger is "no forge.toml
  // present" (forge-action.ts:82-ish); writing this file pre-empts
  // it. The contents declare zero domains, which is the V1 closed-
  // beta default (only forge-moda runs; no music/other vaults).
  const tomlBody = `# Forge vault manifest
# This file declares which domain libraries this vault depends on.
# For V1 closed beta, leave empty — forge-moda is pre-bundled into
# the plugin and available without being declared here.

domains = []

# When v1.1+ ships additional domains (e.g. "music"), add them to
# the list above, e.g.:
# domains = ["music"]
`;

  await adapter.write(tomlPath, tomlBody);
  console.log('Forge: created stub forge.toml');
}
```

Wire it into `runFirstRunCheck` alongside the existing extraction
call. Same independent-of-sentinel semantics as
`ensureBundledForgeModa`: covers both fresh vaults and upgrade-
without-toml cases.

3. Reload Obsidian.
