# v0.2.26 — resolve subdir snippet references (plugin snippet_id + engine caller-scoped bare resolution)

## Why this prompt exists

v0.2.25 shipped the `forge-music/blues/` bundle correctly to the user's
vault. User extracted the bundle, found 8 files in
`~/forge-vaults/test1/forge-music/blues/`, opened `song.md`, clicked
**Forge**. Result:

```
PythonError: SnippetResolutionError: Snippet 'song' not found.
  Searched: authoring, forge-moda, forge-music, forge (built-in).
```

Triage identified **two distinct bugs** rooted in the same blind spot
(library-vault subdirectories were never wired through the snippet-ID
plumbing):

### Bug A — plugin derives `snippet_id` from basename, not path

`src/main.ts:1543` (in `runSnippet`):

```typescript
const snippetId = view.file.basename;
```

For `<vault>/forge-music/blues/song.md`, basename is `"song"`. The
engine's registry indexes `forge-music`'s contents via `os.walk`
(see `snippet_registry.py:168-178`), producing bare IDs `form`,
`twelve_bar_blues_progression`, `blues/song`, `blues/chorus`, etc.
**There is no bare `song` anywhere.** The resolver walks
authoring → forge-moda → forge-music → builtin looking for `song` and
misses on all four.

The fix: when the file lives inside a library-vault subdirectory
(detected by the existing `libraryDirNames()` Set in main.ts), the
snippet_id must be qualified — `"forge-music/blues/song"` — so the
resolver's `/`-branch hits `registry.get_in_vault("forge-music",
"blues/song")` directly.

### Bug B — engine bare lookup ignores caller's scope

Even with Bug A fixed, `song.md`'s body is:

```python
def compute(context):
    chorus1 = context.compute("chorus")
    chorus2 = context.compute("chorus")
    solo = context.compute("solo_chorus")
    chorus3 = context.compute("chorus")
    return sequence(chorus1, chorus2, solo, chorus3)
```

`context.compute("chorus")` calls `self._resolver.resolve("chorus")`
(executor.py:135). `GraphResolver.resolve` sees no `/`, calls
`registry.get_bare("chorus")`, which walks vaults' top-level bare
keys — and `forge-music`'s top-level bare keys are `form` and
`twelve_bar_blues_progression` (the rest are nested under `blues/`).
Miss on all four vaults. `SnippetResolutionError` again.

The fix: `context.compute("chorus")` from a snippet whose qualified
ID is `forge-music/blues/song` should try `forge-music/blues/chorus`
FIRST (sibling-in-same-dir lookup), then fall back to the
resolution-order walk. The `caller_id` is already plumbed into
`ForgeContext.__init__` (executor.py:108-117) for edge tracking and
frozen-snapshot reads — it just isn't passed into the resolver. Both
the `GraphResolver` and `ForgeContext.compute` need the plumbing.

### One bug or two

These are two separate bugs but they share a root cause and the user
can't smoke the fix for Bug A without Bug B also fixed (clicking
Forge on `song.md` immediately fans out via `context.compute`). One
unified prompt is the fastest path to unblock end-to-end smoking the
blues-bundle UX.

Per cowork-protocol §111 (cross-repo phasing): this prompt spans
forge AND forge-client-obsidian. Order of operations: engine fix
(forge) → engine bundle copy into plugin (forge-client-obsidian) →
plugin snippet_id fix (forge-client-obsidian) → tests at each layer
→ version bump v0.2.26 → release.

#### Engine fix — `forge/core/graph_resolver.py`

Replace the body:

```python
from typing import Optional
from forge.core.snippet_registry import SnippetRegistry, BUILTIN_VAULT
from forge.core.exceptions import SnippetResolutionError


class GraphResolver:
  def __init__(self, registry: SnippetRegistry):
    self._registry = registry

  def resolve(self, snippet_id: str, caller_id: Optional[str] = None) -> dict:
    """Resolve a snippet reference per ADR 0002. Raises SnippetResolutionError on miss.

    When `caller_id` is provided AND it's a qualified ID with a subdir
    component (e.g. 'forge-music/blues/song'), bare references are
    probed against the caller's own directory FIRST. This makes
    `context.compute("chorus")` inside `forge-music/blues/song.md`
    resolve to `forge-music/blues/chorus` rather than missing all
    vaults. Falls back to the legacy bare walk on miss.
    """
    hit = self._lookup(snippet_id, caller_id)
    if hit is None:
      raise SnippetResolutionError(reference=snippet_id, searched=self._searched_for(snippet_id))
    return hit

  def try_resolve(self, snippet_id: str, caller_id: Optional[str] = None) -> Optional[dict]:
    """Non-raising variant for callers that want to inspect on a miss."""
    return self._lookup(snippet_id, caller_id)

  def _lookup(self, snippet_id: str, caller_id: Optional[str] = None) -> Optional[dict]:
    if "/" in snippet_id:
      vault_name, bare = snippet_id.split("/", 1)
      return self._registry.get_in_vault(vault_name, bare)
    # Bare ref. Try caller-scoped sibling lookup first when applicable.
    if caller_id is not None and "/" in caller_id:
      caller_vault, caller_bare = caller_id.split("/", 1)
      if "/" in caller_bare:
        caller_dir = caller_bare.rsplit("/", 1)[0]
        sibling = self._registry.get_in_vault(caller_vault, f"{caller_dir}/{snippet_id}")
        if sibling is not None:
          return sibling
    return self._registry.get_bare(snippet_id)

  def _searched_for(self, snippet_id: str) -> list:
    if "/" in snippet_id:
      vault_name, _ = snippet_id.split("/", 1)
      return [vault_name]
    return [self._label(v) for v in self._registry.resolution_order()]

  @staticmethod
  def _label(vault_name: str) -> str:
    if vault_name == BUILTIN_VAULT:
      return "forge (built-in)"
    return vault_name
```

The semantics: caller-scoped sibling is a TRY-FIRST, not a
TRY-ONLY. If the sibling doesn't exist, fall through to the
legacy bare walk. This preserves the current resolution-order
contract for everything that worked pre-fix.

#### Plugin fix — `src/snippet-id-from-path.ts` (new file, pure-core)

```typescript
// v0.2.26 — derive the qualified snippet_id from a vault-relative
// file path. Files inside a library-vault subdirectory (declared by
// the user's `forge.toml` and surfaced via the plugin's
// libraryDirNames() Set) get qualified IDs like
// "forge-music/blues/song"; vault-root files keep the basename
// behavior ("song" for "song.md").
//
// This is the source-of-truth for snippet_id derivation. Any place
// in the plugin that resolved a snippet via basename pre-v0.2.26 is
// a candidate for migration — but the load-bearing call site is
// runSnippet() in main.ts; other view.file.basename uses are
// user-facing message strings, not snippet IDs.

export function snippetIdFromPath(
  filePath: string,
  libraryDirNames: Set<string>,
): string {
  // Strip .md extension if present (caller is expected to gate by
  // file.extension === 'md'; helper is lenient about the suffix).
  const withoutExt = filePath.replace(/\.md$/i, "");
  const firstSlash = withoutExt.indexOf("/");
  if (firstSlash === -1) {
    // Vault root.
    return withoutExt;
  }
  const topDir = withoutExt.slice(0, firstSlash);
  if (libraryDirNames.has(topDir)) {
    // Library subdir — keep the full qualified path.
    return withoutExt;
  }
  // Non-library subdir — fall back to basename (legacy behavior).
  return withoutExt.slice(withoutExt.lastIndexOf("/") + 1);
}
```

## 10. One observation for the v1.0 retrospective

The bundle-extraction code path (v0.2.13 `copyDirRecursive`) was
correct for arbitrary depth, but the SNIPPET-ID code path
(v0.2.x basename derivation, present since v0.2.2 forge-click)
silently assumed flat library vaults. This is the same "the lower
layer is correct but the upper layer's assumption about it is
wrong" failure shape as the v0.2.22→v0.2.23 inputs-threading bug
(JS side was correct; Python side discarded it). Worth flagging
in the v1.0 audit:

**Any cross-layer contract (file path → snippet ID, JS args →
Python kwargs, bundle copy → registry index) needs a test that
exercises the FULL chain end-to-end, not just the individual
layers.** The unit tests for `copyDirRecursive` (preserves nested
structure) passed; the unit tests for the registry's `os.walk`
(indexes subdirs as `blues/song`) passed; but no test wired up the
plugin's `view.file.basename → runSnippet → engine.resolve` chain
to confirm the snippet_id at the seam matched what the engine
expected. The §1.3 case-7 integration test in this prompt
("execute parent calling context.compute('child') in subdir,
assert child returns 42") is the canonical seam test; analogous
seam tests for other contracts are v1.0 audit candidates.
