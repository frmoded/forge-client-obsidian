# v0.2.8 — remove uvicorn auto-spawn (dead code post-v0.2.6)

## Why this prompt exists

v0.2.6 made `connectVault` Pyodide-routed; v0.2.7 silenced the
post-`/generate` `/sync_dependencies` ECONNREFUSED noise. But on the
user's dev machine, killing uvicorn produces this log on every
reload:

```
Forge: Server offline, attempting to spawn...
Forge: Spawning server on port 8000...
```

Tracing it: `main.ts:480` calls `ensureServerRunning(serverUrl)` in
`onload`. `ensureServerRunning` (server.ts:263) GETs `serverUrl/test`,
catches the connection refusal, and calls `spawnForgeServer`
(server.ts:276), which `spawn`s uvicorn using a **hardcoded** path:

```typescript
const pythonPath = '/Users/odedfuhrmann/projects/forge/.venv/bin/python';
spawn(pythonPath, ['-m', 'uvicorn', 'forge.api.server:app', '--port', port], ...)
```

This is pre-V1 dev-convenience code. Three problems:

1. **Hardcoded venv path** — works only on the original developer's
   machine; fails silently on any other machine (the spawn errors
   are swallowed by `spawn`'s default error handling, not the
   try/catch in `ensureServerRunning`).
2. **Defeats the v0.2.6 closed-beta promise on dev machines** —
   uvicorn keeps respawning seconds after manual kill, making the
   "no uvicorn" smoke untestable on the developer's own laptop.
3. **Dead code post-v0.2.6** — `connectVault` is Pyodide-routed,
   `computeSnippet` is Pyodide-routed. The only remaining HTTP
   callers (`/sync_dependencies`, `/canonicalize`, `/freeze`) are
   engine-side optional and either non-fatal (v0.2.7 dep-sync) or
   menu-only (canonicalize, freeze). Nothing on the click path
   needs uvicorn.

Scope: rip out the auto-spawn + heartbeat code, remove call sites,
drop the unused `spawn` import, bump version, ship.

## 1. Remove the plugin-init auto-spawn

`src/main.ts:480`:

```typescript
await runFirstRunCheck(this.app);
ensureServerRunning(this.settings.serverUrl);  // ← DELETE THIS LINE
```

Just delete the line. No replacement needed — the Pyodide host
initialization already happens elsewhere in `onload` and that's the
only "server" the V1 plugin needs.

## 2. Remove the install-flow auto-spawn

`src/forge-action.ts:245`:

```typescript
try {
  await ensureServerRunning(host.serverUrlOf());  // ← DELETE
  const vaultPath = host.vaultPathOf();
  await connectVault(host.serverUrlOf(), vaultPath);
  const res = await computeSnippet(
    host.serverUrlOf(), vaultPath, 'install', [vaultName, version]);
```

`connectVault` and `computeSnippet` both have Pyodide routes now,
so the spawn-prelude is unnecessary. Delete the line; leave the
subsequent `connectVault` + `computeSnippet` calls alone.

## 3. Delete the dead helpers in server.ts

Remove the following from `src/server.ts`:

- `pingServer` (lines 253-261) — exported, never called. Pure cruft.
- `ensureServerRunning` (lines 263-274) — only callers being deleted
  in §§1-2.
- `spawnForgeServer` (lines 276-end) — only caller is
  `ensureServerRunning`.
- The `import { spawn } from 'child_process';` at line 2 — only
  used by `spawnForgeServer`.

Also drop `ensureServerRunning` from the import lists in:
- `src/main.ts:14`
- `src/forge-action.ts:3`

## 4. Verify nothing else references the deleted symbols

Run these checks after the edits:

```bash
grep -rn 'ensureServerRunning\|pingServer\|spawnForgeServer' src/
grep -rn "from 'child_process'" src/
```

Both should return empty (no hits). If anything remains, audit and
either re-route through Pyodide or document why HTTP-only is OK
for that call site.

## 5. Test addition

Add a regression test to `src/main.test.ts` (or whatever test file
already exercises the plugin class — create `closed-beta-ux.test.ts`
sibling if no main test file exists) that asserts plugin
initialization doesn't spawn child processes:

```typescript
import { test } from 'node:test';
import assert from 'node:assert';

test('plugin init does not spawn external processes', async () => {
  // Verify the bundled main.js does not import child_process.
  // This is the load-bearing assertion — main.ts dropping the import

```

## 8. Manual smoke guidance for user

The v0.2.8 ship gate is the same smoke that v0.2.7 couldn't prove
on the dev machine:

1. **Install v0.2.8.** Drag-and-drop replacing v0.2.7 folder. Verify
   `manifest.json` shows 0.2.8.
2. **Kill uvicorn.** `kill $(lsof -tiTCP:8000 -sTCP:LISTEN)`. Verify
   empty `lsof -iTCP:8000 -sTCP:LISTEN -P`.
3. **Reload Obsidian** (Cmd+P → "Reload app without saving"). Watch
   the console.
4. **Confirm no auto-spawn.** The console should **NOT** contain:
   - `Forge: Server offline, attempting to spawn...`
   - `Forge: Spawning server on port 8000...`
5. **Confirm uvicorn stays dead.** Wait 5 seconds, re-run `lsof -iTCP:8000 -sTCP:LISTEN -P`.
   Expect: empty. (On v0.2.7 it would have come back via auto-spawn.)
6. **Forge-click smoke from v0.2.7 §4.4.** Clear console, Forge-click
   a snippet whose Python facet was deleted. Expect:
   - α call succeeds (Python writes back).
   - `info`-level "dependency sync skipped" line fires **once**.
   - Compute renders in Forge Output.
7. **Second Forge-click.** Should produce no additional dep-sync
   log line. Once-per-session guard holds.

## 9. Deviations

Standard "deviations" section. Specific things to call out:

  (e.g., via a transitive dep that uses it), the symbol-grep tests

  `computeSnippet`, no engine-side state needed.
