# Smoke vault snippets — `random_name` + `hello_random` for freeze-edge demo in `smoke-v0.2.13`

## Scope

Create two new snippets in the user's `smoke-v0.2.13` vault that together exercise edge freezing in a tiny, contained way:

1. **`random_name`** — takes `n` as input, returns a string of `n` random lowercase letters.
2. **`hello_random`** — calls `random_name(n=5)` then calls `greet(name=...)` with the result.

Pre-freeze: each Forge-click of `hello_random` prints "Hello <different-5-letters>". Post-freeze (user freezes the edge `hello_random` → `random_name`): each Forge-click prints "Hello <same-5-letters>". That contrast is the freeze demo.

What this prompt does NOT do:
- Touch any repo. Both snippets live in the user's vault on disk; no commits, no version bumps, no plugin release.
- Add tests to `forge/tests/vault/` — the snippets are smoke-vault content, not canonized test fixtures. (If the user later wants them canonized, that's a separate drain.)
- Build the freeze smoke checklist itself — that lands in chat AFTER cowork reviews this drain.

## Why

Two threads converge here:

1. **Edge freezing is shipped in v0.2.30 (route /freeze through Pyodide) but lacks a tiny user-facing demo snippet to validate the round trip.** Forge-music has been using freeze indirectly via the music demo; a dedicated micro-demo in the smoke vault gives the user a clean test surface separate from music-domain dependencies.
2. **`hello_random` exercises `context.compute('random_name', n=5)`** — the same kwargs-through-compute path that the v0.2.22-v0.2.23 plumbing arc fixed and that Loom (v0.2.36) is currently the only shipped exerciser of in forge-music. Having a second exerciser in a non-music smoke vault is independent regression coverage.

## Implementation notes

### `random_name.md` content

```markdown
---
type: action
inputs: [n]
description: Returns a string of n random lowercase letters. Smoke-vault demo snippet for edge freezing.
---

# English

Returns a string of `n` randomly-chosen lowercase letters (a-z). Used as the upstream
caller in the freeze-edge demo — pre-freeze each call gives a different string;
post-freeze the call returns the snapshot-pinned value.

---

# Python

```python
import random
import string

def compute(context, n):
    return ''.join(random.choices(string.ascii_lowercase, k=n))
```
```

### `hello_random.md` content

```markdown
---
type: action
description: Calls greet with a 5-letter random name. Smoke-vault demo for edge freezing.
---

# English

Generates a 5-letter random name via `random_name(n=5)`, then calls `greet` with it.
Pre-freeze: prints "Hello <random>" with a fresh name each Forge-click. Post-freeze
the edge `hello_random → random_name`: prints "Hello <same-name>" repeatedly until
unfrozen.

---

# Python

```python
def compute(context):
    name = context.compute('random_name', n=5)
    context.compute('greet', name=name)
```
```

### Backtick-in-docstring caveat

Neither snippet's Python uses backticks inside docstrings/comments, so the v0.2.20/v0.2.23 template-literal trap doesn't apply here. (Flagging because it's CC's protocol checklist item.)

## Don'ts

- **Don't create the vault if missing.** Pause to `questions/` instead. Creating the vault from scratch would lose the v0.2.13-era context (existing greet.md, any other snippets, vault forge.toml settings).
- **Don't modify `greet.md`.** It's an existing fixture; the new snippets adapt to its signature, not the other way around.
- **Don't add a `seed` parameter to `random_name`** to make it deterministic. The whole point of the demo is that it's non-deterministic pre-freeze; freezing makes it deterministic. Adding a seed defeats the demo.
- **Don't bundle these snippets into a plugin release.** They're user-vault content for smoke testing, not distributable artifacts.
- **Don't write the freeze smoke checklist.** Cowork owns that — it lands in chat after CC's feedback is reviewed.
