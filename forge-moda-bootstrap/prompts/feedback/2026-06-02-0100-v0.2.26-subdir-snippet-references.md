```python
def resolve(self, snippet_id: str, caller_id: Optional[str] = None) -> dict:
    hit = self._lookup(snippet_id, caller_id)
    if hit is None:
      raise SnippetResolutionError(reference=snippet_id, searched=self._searched_for(snippet_id))
    return hit

def try_resolve(self, snippet_id: str, caller_id: Optional[str] = None) -> Optional[dict]:
    return self._lookup(snippet_id, caller_id)

def _lookup(self, snippet_id: str, caller_id: Optional[str] = None) -> Optional[dict]:
    if "/" in snippet_id:
      vault_name, bare = snippet_id.split("/", 1)
      return self._registry.get_in_vault(vault_name, bare)
    # Bare ref. Try caller-scoped sibling lookup first when applicable
    # (v0.2.26). TRY-FIRST, not TRY-ONLY — misses fall through to the
    # legacy resolution-order walk below.
    if caller_id is not None and "/" in caller_id:
      caller_vault, caller_bare = caller_id.split("/", 1)
      if "/" in caller_bare:
        caller_dir = caller_bare.rsplit("/", 1)[0]
        sibling = self._registry.get_in_vault(caller_vault, f"{caller_dir}/{snippet_id}")
        if sibling is not None:
          return sibling
    return self._registry.get_bare(snippet_id)
```

```typescript
export function snippetIdFromPath(
  filePath: string,
  libraryDirNames: Set<string>,
): string {
  const withoutExt = filePath.replace(/\.md$/i, "");
  const firstSlash = withoutExt.indexOf("/");
  if (firstSlash === -1) return withoutExt;
  const topDir = withoutExt.slice(0, firstSlash);
  if (libraryDirNames.has(topDir)) return withoutExt;
  return withoutExt.slice(withoutExt.lastIndexOf("/") + 1);
}
```

## Protocol comments for driver

1. **The §2 audit incompleteness was a real shape lesson.** I wrote the prompt with an explicit grep-based audit naming 5 lines — and the audit missed 3 additional snippet_id call sites because lines 1227 + 1498 weren't in the original grep range and line 762 was confused with 763. The lesson: **a "grep at prompt-authoring time" audit captured in the prompt body is a snapshot; CC should re-run the same grep mid-drain before declaring scope complete.** Worth adding to cc-prompt-queue.md: any prompt that names specific line numbers as the "complete set of call sites" must include a re-grep step in §1.3, not just trust the prompt's own enumeration. The §80 fixture-drift rider has the same shape ("verify the claim mechanically") and could be widened to cover call-site enumerations too.

2. **The engine-bundle copy step worked smoothly but should be automated.** This is the third drain in two days touching `forge/core/*.py` (v0.2.17 added `refresh_file`, the constitution-V4 work, this prompt). Each time it's a manual `cp`. Either:
   - **Build-time sync** — `npm run build` runs a sync script that copies `~/projects/forge/forge/core/*.py` into `assets/engine/forge/core/` before bundling. Build fails if the copy isn't a no-op (someone manually edited the bundled engine).
   - **Test-time drift check** — a new test reads both `~/projects/forge/forge/core/graph_resolver.py` and `assets/engine/forge/core/graph_resolver.py`, asserts byte-equal. Fails the suite when they drift.
   The drift check is cheaper (no build-script changes) and catches the failure mode where someone bumps forge but forgets to copy the bundle. Recommend queuing as a separate prompt.

3. **The integration drift-protection assertion (plugin test case 8) paid off.** That assertion reads main.ts at test-start and grep-matches for `snippetIdFromPath(view.file.path, this.libraryDirNames())`. When I migrated the THREE other snippet_id call sites, I needed to ensure the helper was actually wired in everywhere — but the assertion only checks `runSnippet`. The other three (generate, canonicalize, syncEdgesForActive) aren't drift-protected. For a future revision, either widen the assertion to grep for ALL snippet_id call sites that should use the helper, or extract a helper-wrapper at the main.ts level (`this.deriveSnippetId(view.file)`) so there's only one call site to drift-check. Recommend the latter for v0.2.27+.

4. **Cross-repo two-layer fix in one drain worked.** Per cowork-protocol §111 phasing guidance, this prompt explicitly bundled engine + plugin into one prompt with phased sections (engine fix → bundle copy → plugin fix → tests at each layer → version bump → release). The total drain was ~25 minutes. Splitting into two prompts (engine first, plugin second) would have added ~10 minutes of context-restoration overhead and prevented end-to-end test 7 (`test_context_compute_threads_caller_id_through`) from being meaningful in the engine-only drain (it would need plugin migration to land before becoming load-bearing). **Recommendation: keep "one unified prompt" as the default for tightly-coupled two-layer fixes; split only when the layers can land sequentially with independent value.**

5. **The "lower-layer-correct, upper-layer-assumption-wrong" pattern surfaced again, third time in the v0.2.x arc.** Per §10 of the prompt: the bundle-extraction code was correct for arbitrary depth (v0.2.13 `copyDirRecursive`); the registry's `os.walk` correctly indexes subdir snippets (snippet_registry.py:168-178 unchanged); but the seam where the plugin sends a snippet_id to the engine assumed flat-vault structure. Analogous to v0.2.22→23 (JS layer correct, Python layer dropped inputs) and v0.2.18 (memfs sync correct, hook ordering wrong). The v1.0 audit candidate "every cross-layer contract needs a seam test" remains the right framing; test 7 here (`test_context_compute_threads_caller_id_through`) is the canonical seam-test shape — it doesn't unit-test the layers, it runs them together.
