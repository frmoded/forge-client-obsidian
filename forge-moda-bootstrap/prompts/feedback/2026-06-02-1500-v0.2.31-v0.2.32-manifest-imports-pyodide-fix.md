# v0.2.31 + v0.2.32 — Pyodide library-vault scan fix (post-mortem)

The MEMFS mount was correct — all 8 blues files present in /bundle/user-vault/forge-music/blues/. Only the empty authoring vault made it into `_vaults`.

## §Fix (v0.2.32)

`forge/core/manifest.py` had three unconditional top-level imports incompatible with closed-beta Pyodide:

| Import | Status in Pyodide | Used by |
| --- | --- | --- |
| `tomli_w` | Not vendored | `write_manifest` only |
| `packaging.version` | Not vendored | `_validate_semver` only |
| `forge.installer.exceptions` | `forge/installer/` excluded from engine bundle by drift filter | `ValidationError` raises throughout |

`read_manifest` (the only function the plugin exercises) doesn't need tomli_w, only uses `_validate_semver` indirectly via `_from_dict → _validate`, and uses `ValidationError` for raising. So fix shape:

- `tomli_w`: lazy-import inside `write_manifest`.
- `packaging.version`: `_semver_validators()` helper returns a regex-based fallback `(Version, InvalidVersion)` pair when packaging isn't on sys.path. Validation strictness drops slightly (regex `^\d+(\.\d+){1,2}([-+][0-9A-Za-z.\-+]+)?$` vs PEP 440) but obviously-bad version strings still get caught.
- `ValidationError`: try-from-`forge.installer.exceptions`, fallback to local `class ValidationError(Exception): pass`. Callers' `except ValidationError` still works.

CPython behavior unchanged. Engine 429/4 skipped.

### 2. `DOMAIN_AVAILABILITY` fail-loud registry (v1.0 audit)

Today is the **second time** the same "silent-skip on ImportError" pattern hid a real bug:

- **v0.2.27**: `executor.py`'s `try: import music21; except ImportError: _MUSIC21_NAMES = {}` silently degraded the music domain to empty. Surfaced 6 weeks later when a user forge-clicked a music snippet → NameError six call-levels deep.
- **v0.2.32**: `snippet_registry._scan_library_vault`'s `try: read_manifest; except Exception: self.errors.append(...)` silently dropped the library. Surfaced today when the user forge-clicked → SnippetResolutionError. Errors list populated but never surfaced to JS console.

**Both have the same anti-pattern shape**: catch a fatal-to-the-feature ImportError, downgrade to "feature silently absent," return None / empty dict, let runtime code discover the gap. Fix one root cause and both go away.

Proposed v1.0 design (per v0.2.27 retro):

```python
DOMAIN_AVAILABILITY = {}  # populated at engine boot
for domain_name, requires in DOMAIN_REQUIRES.items():
    DOMAIN_AVAILABILITY[domain_name] = all(_can_import(m) for m in requires)
# At vault-load time, if vault declares domains=["music"] but
# DOMAIN_AVAILABILITY["music"] is False, emit a load-time error
# pointing at the missing wheel — not a runtime NameError or
# SnippetResolutionError six call levels deep.
```

Extend the same pattern to the registry: surface library-scan errors at boot, not at lookup time. Either log them via the JS console bridge, or fail-loud on `domains = [...]` declarations whose libraries failed to scan.

### 7. NEW: forge.core.* imports auditable for Pyodide compat

This drain's root cause was an unconditional import in `forge.core.manifest` that didn't exist in Pyodide. The fix made it lazy; what would have caught it pre-shipping?

**Suite-time check**: scan every `.py` file under `assets/engine/forge/core/` (and `moda/`, `music/`) for top-level `import` statements; check each module name against a Pyodide-available allowlist. Flag any unconditional import of a module not in the allowlist.

Allowlist members: stdlib + numpy + pyyaml + (the 11 vendored wheels' top-level names: music21, chardet, jsonpickle, more_itertools, webcolors, joblib, requests, urllib3, certifi, charset_normalizer, idna).

Anything else under `try: ... except ImportError:` is fine. Unconditional imports of non-allowlist modules fail the check.

This would have caught `import tomli_w` + `from packaging.version import ...` at engine-bundle-sync time. Cheaper than the new v0.2.32 Pyodide-warm test in computer time (no Pyodide boot), broader in coverage (every .py file, not just the ones we explicitly test).

Recommended next-up infrastructure prompt alongside item 1.

## §Protocol comments (from this drain)

1. **The v0.2.30 engine-bundle drift check passed at release-zip time — and the bundle WAS byte-equal to source.** But the bug wasn't drift; it was an environment delta between the suite tests (which boot Pyodide with the vendored wheels) and the production registry-scan path (which exercises `_scan_library_vault` against forge.toml-bearing dirs). The drift check is necessary but insufficient. Today's regression test (item #7's smaller cousin) is the load-bearing protection going forward.

2. **The v0.2.31 diagnostic-only release pattern worked cleanly.** Diagnose → ship instrumentation → user reproduces with instrumentation → diagnosed. Total wall-clock from crash report to root cause: ~15 minutes. Ship pattern worth codifying: "for any production-runtime bug that doesn't reproduce in suite, ship a small instrumented release ASAP rather than guessing." The instrumentation gets removed in the fix release.

3. **The FORGE-DEBUG prefix convention was useful** — easy to grep, easy to ask the user to paste only those lines, easy to remove. Worth keeping as a standing convention for ad-hoc instrumentation.

4. **The "lazy import with fallback" pattern in manifest.py is reusable** for any future forge.core.* module that needs to work in Pyodide. The try/except shim for `ValidationError` is particularly worth keeping in mind — when an engine module imports from `forge.installer.*`, the lazy-fallback shim is the cheap fix.

5. **The Pyodide-warm regression test boot cost (~3s) is acceptable** when reused via the singleton pattern from `music21-bundle.test.ts`. Adding the one new test cost ~2s incremental suite time. Future engine boundary changes should default to adding a Pyodide-warm test rather than relying on CPython-only verification.

## §10 v1.0 retrospective observation

**Same silent-skip anti-pattern, third drain.** v0.2.27 (music21 missing), v0.2.32 (manifest imports missing). Both caught by "user can't compute snippet" surfaced 6+ weeks late. The third was forge-music vault scan, same root cause family.

The v1.0 audit item is concrete now: **eliminate the silent-skip pattern across all engine code paths**. Specifically:

- `try: import X; except ImportError: <empty dict>` → fail-loud at module load with actionable error, OR populate a `_AVAILABILITY` registry checked at vault-load time.
- `try: read_manifest; except Exception: errors.append()` → either bubble the errors to a known surface (JS console, plugin notice), or fail-loud at scan time.
- `from forge.installer.* import` in `forge.core.*` → either move the imported symbol into `forge.core.*` or add the suite-time scope-boundary audit (item #7).

Three concrete production bugs in three weeks; one fix would have caught all three. Worth promoting from "v1.0 audit" to "next infrastructure drain."
