# v0.2.15 — bundle forge-music into the plugin (mirror forge-moda)

## 1. Bundled forge-music content

### Bundling mechanism

Confirmed by inspection: **forge-moda is hand-mirrored** into
`forge-client-obsidian/assets/vaults/forge-moda/` and checked into
git. `scripts/copy-assets.mjs` regenerates `assets/manifest.json`
from disk contents on every build (so files appearing there auto-
appear in the manifest), but the **actual file copy from
`~/projects/forge-moda/` is manual** — there's no automated sync.
`scripts/build-release-zip.mjs:58` asserts the bundle's presence
(`assets/vaults/forge-moda/forge.toml`) as a preflight check.

Replicated identically for forge-music:

- Copied `~/projects/forge-music/{forge.toml, form.md,
  twelve_bar_blues_progression.md}` → `assets/vaults/forge-music/`.
- Excluded `LICENSE`, `NOTICE`, `README.md` (forge-moda excludes
  the equivalents — only snippets + forge.toml + optional `_meta/`
  go in the bundle).
- forge-music has no `_meta/` dir in source, so nothing additional
  to copy.
- Added preflight asserts in `build-release-zip.mjs` for
  `assets/vaults/forge-music/forge.toml` and
  `assets/vaults/forge-music/form.md`.

### Policy implication

forge-music content updates (new snippets, edits to existing) will
require a manual re-sync into `assets/vaults/forge-music/` before
each release. Same policy as forge-moda. The build-zip preflight
catches "I forgot to copy" cases but doesn't catch "I edited the
source but not the bundle" drift.

A scripted sync (npm script that copies `~/projects/forge-music/*`
→ `assets/vaults/forge-music/` with the exclusion filter) is a
trivial follow-up. Flagged as v1.1+ thinking rather than v0.2.15
scope.

Gates on the user's `forge.toml` declaring `"music"` in the

skips comments (`#`), and returns true only for an active
`domains\s*=\s*\[...]` line that includes the exact quoted
`"music"` string.

**Seventh pure-core extraction in this arc.** See §11 below.

## 12. One observation

The hand-sync count for bundled-library identity has reached
**three places** (JS-side `BUNDLED_LIBRARY_NAMES`, Python-side
`_BUNDLED_LIBRARIES_V1`, and forge-action.ts `BUNDLED_VAULTS`).
Plus the welcome.ts gate logic implicitly depends on the same
identity. Forgetting one on a future bundle addition (e.g., a
v1.1 forge-puzzle) would manifest as silent skips — content
mounted in MEMFS but no extraction, or extraction without resolver
registration.

**Recommended v1.0 audit task**: one shared `BUNDLED_VAULTS_V1`
constant exported from a small pure-core file
(e.g. `src/bundled-vaults.ts`), imported by:
- `forge-action.ts` (installVault gate).
- `welcome.ts` (looping over the set for extraction per declared
  domain).
- `pyodide-host.ts` JS side (the Set construction).
- Build-time Python generator script that writes the
  `_BUNDLED_LIBRARIES_V1 = [...]` literal into the embedded Python
  block, OR a JS↔Python boundary call that sets the list from JS.

Logging the suggestion; not in v0.2.15 scope. Closed-beta closes
out of the v0.2.x arc cleanly even with three copies — the cost
of inconsistency is borne by the next bundle-addition prompt.
