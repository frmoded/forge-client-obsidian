## Deviations + observations

### Two surprises that shifted approach mid-drain

1. **micropip + `deps=False` does NOT skip network calls.** I started by using `await micropip.install("emfs:/bundle/wheels/music21-...whl", { deps: false })` — the prompt suggested it. Test 1 passed locally but ran in 32.7s and the verbose output revealed CDN downloads. With the cache cleared, the same install pulled matplotlib + requests + pillow + 14 other packages from `cdn.jsdelivr.net/pyodide/v0.29.4/full/`. Even with `deps: false`, micropip parses the wheel's METADATA Requires-Dist and routes recognized PyPI/pyodide-stock packages through `loadPackage` → indexURL → CDN. **Conclusion: micropip is fundamentally not suitable for closed-beta no-network production**, regardless of the `deps` flag. Pivoted to stdlib `zipfile` extraction + sys.path manipulation. Documented at length in pyodide-host.ts:225-234.

2. **music21's __init__.py eagerly imports `requests`.** Music21's __init__:191 does `from music21.test.testRunner import mainTest` (test code in __init__!), which transitively pulls `music21.converter`, which `import requests` at line 50. So even just `import music21` fails without requests. Initial plan was to vendor only music21 + the 5 declared deps; actual minimum set is 11 wheels (added requests + urllib3 + certifi + charset-normalizer + idna). Documented in the prompt's §1.3 was "find minimum dep set"; actual hunt took 3 failed-import iterations.

### Skipped from prompt's §1.3 — intentional

The prompt suggested setup-assets.mjs gets updated to "extend the assertion to include the new wheels." On reading the script, the assertion is only for Pyodide stock packages — wheels under `assets/wheels/` are picked up by build-manifest.mjs's category walker automatically. No setup-assets.mjs change needed. Build verifies: `wheels: 11 files` line in the build output confirms the wheels manifest is populated.

The strict failure mode (boot-time `runPython("import music21; from music21 import key, stream")` verification) is implicit via the zipfile-extract block + first-use behavior: if a wheel is missing the user sees the import failure at host init, not six call-levels deep. ## Protocol comments for driver

1. **"micropip is fine for closed-beta" is now demonstrably false.** The cc-prompt-queue.md / cowork-protocol mentions micropip in passing as the wheel-install path; v0.2.27 proved that doesn't survive contact with the no-network production constraint. Worth adding a one-paragraph protocol rider somewhere visible: *"In closed-beta-with-no-network constraints, micropip's loadPackage callback path is a hidden network dependency even with `deps=False`. Prefer stdlib zipfile + sys.path manipulation when vendoring wheels."* If a future engine wants a dynamically-managed package (something that gets installed from a user-provided wheel post-deployment), micropip is still the right answer — but for boot-time stock wheels, zipfile is simpler and provably correct.

2. **The prompt's "investigate minimum dep set first" instruction (§1.3) was load-bearing.** Without it, CC would have shipped 6 wheels and discovered the requests-needed error post-release. Pattern worth codifying: *"Bundling external dependencies in a constrained-runtime environment requires an iterative top-level-import probe (import → catch ModuleNotFoundError → vendor missing → repeat). Don't trust the declared dep list; modules vary in how aggressively they eager-import."* Add to cc-prompt-queue.md alongside the §80 fixture-drift rider.

3. **The §80 drift-protection rider's spirit applied here non-trivially.** The test fixture extracts wheels from `assets/wheels/` at test-start (same place production reads from at boot). If a future drain adds a wheel but forgets to vendor it in `assets/wheels/`, the test fails before anything ships. Drift protection mechanically rather than via comment-NOTE. Good shape.

4. **One unified TDD-hard-rule drain handled the "boot-time + first-use" two-phase verification cleanly.** Test 1 verifies boot (`import music21` works) — fast-fails on missing wheel. Test 4 verifies first use (form.md-equivalent compute) — fast-fails on missing music21 submodule. Together they cover the full closed-beta boot-to-first-use path. The §141 "push every assertion into the suite" rule paid off — without Pyodide-warm tests, the user's smoke would have been the only verification.

## §10 One observation for the v1.0 retrospective (expanded from prompt)

The prompt's §10 named the "silent-degradation domain bundle" pattern in executor.py. v0.2.27 surfaced TWO concrete instances simultaneously:

- `_MUSIC21_NAMES` was `{}` because music21 wasn't bundled (the user's NameError).
- `_FORGE_MUSIC_LIB_NAMES` would have been `{}` even after music21 landed because `forge.music.lib` wasn't in the engine bundle either. The two failures stacked: even after vendoring music21, you'd hit `NameError: name 'sequence' is not defined` because `_FORGE_MUSIC_LIB_NAMES` was empty.

I caught both by copying `forge/music/{__init__,lib}.py` into the engine bundle as part of the same drain. But if the prompt hadn't named the engine-bundle audit as §2, the second gap could easily have been missed.

**v1.0 fail-loud DOMAIN_AVAILABILITY registry** (per prompt §10) should distinguish:
- "Domain X declared but not buildable in this environment" (e.g., music21 missing in Pyodide → music domain unavailable).
- "Domain X's helper bundle is buildable but engine helpers are missing" (e.g., music21 present but forge.music.lib missing from sys.path).

Both should fail-loud at vault-load time with actionable error messages naming the missing piece. The current silent `_DOMAIN_GLOBALS["music"] = {**_MUSIC21_NAMES, **_FORGE_MUSIC_LIB_NAMES}` pattern collapses both into "empty dict" with no signal.

**Adjacent v1.0 item**: an engine-bundle drift check. The `assets/engine/forge/` tree in the plugin repo and the `~/projects/forge/forge/` tree in the engine repo have to stay byte-equal for the music + moda + core domains. Three manual-cp cycles in two days (v0.2.17, v0.2.26 [executor + graph_resolver], v0.2.27 [music/__init__ + lib]). Either a sync script that runs in CI, a test-time string-diff check, or a manifest-of-engine-files audit. Cheapest is the test-time string-diff; cheapest with the strongest signal is the sync script in CI. v1.0 audit candidate.
