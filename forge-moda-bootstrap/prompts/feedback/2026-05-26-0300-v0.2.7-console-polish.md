# v0.2.7 — console polish: silence post-/generate ECONNREFUSED + first-run welcome

## 1. Silence the sync_dependencies warn on network refusal

Extracted the predicate to a pure-core helper so node --test can
exercise it without an Obsidian shim:

`src/closed-beta-ux.ts`:
```typescript
export function isNetRefusalError(msg: string): boolean {
  return /ERR_CONNECTION_REFUSED|ECONNREFUSED|ENOTFOUND|ENETUNREACH/i.test(msg);
}
```

Catch block in `main.ts:1247-1267` (was 1245-1251):

```typescript
try {
  await syncDependencies(this.settings.serverUrl, vaultPath, id);
} catch (e) {
  const msg = e instanceof Error ? e.message : String(e);
  if (isNetRefusalError(msg)) {
    if (!this.b7SyncSkippedLogged) {
      console.info(
        'Forge: dependency sync skipped — no local engine reachable at '
        + `${this.settings.serverUrl}. This is expected in closed-beta; `
        + 'dependency-section refresh is engine-side only.',
      );
      this.b7SyncSkippedLogged = true;
    }
  } else {
    // Non-network failure — keep the warn so real bugs surface.
    console.warn(`Forge: sync_dependencies failed for '${id}'`, e);
  }
}
```

Field declared at line 190 alongside the other private state:
```typescript
private b7SyncSkippedLogged = false;
```

`syncEdgesForActive` (the explicit user-invoked path at
`main.ts:1255-`) was deliberately left unchanged — that command
SHOULD loudly fail if the engine isn't reachable.

## 2. First-run welcome notice

Persistence: added `seenWelcome: boolean` to `ForgeSettings` so it
piggybacks on the existing `loadData`/`saveData` flow rather than
introducing a sibling state file. Default `false`; `Object.assign`
in `loadSettings` preserves the default when the field is absent
(v0.2.6 → v0.2.7 migrators see the notice once).

Message composition extracted to `welcomeMessage(hasToken)` in
`closed-beta-ux.ts` for pure-unit-testing. Two branches:

- **No token** (closed-beta typical): "Welcome to Forge. To enable
  AI-powered snippet generation, paste your transpile token at
  Settings → Forge → Transpile service → Transpile service token."
- **Token set**: "Forge is ready. Open Settings → Forge to review
  your configuration."

Wired in `main.ts` at the end of `onload()` (after
`runFirstRunCheck` + `ensureServerRunning`, after Pyodide host wire,
after settings load — i.e., after all setup has settled):

```typescript
if (!this.settings.seenWelcome) {
  const hasToken = !!this.settings.transpileServiceToken?.trim();
  new Notice(welcomeMessage(hasToken), 10000);  // 10s vs default 5s
  this.settings.seenWelcome = true;
  await this.saveSettings();
}
```

Added `onunload()` to reset `b7SyncSkippedLogged = false` per the
prompt — explicit reset documents intent even though the class
re-instantiation on plugin reload would handle it.

## 10. One observation

Three releases in a row (v0.2.5, v0.2.6, v0.2.7) have been
incremental closed-beta polish driven by user smoke. That cadence
is fine for closed beta — small focused fixes ship cleanly — but
the cumulative ergonomic-fix surface area is starting to deserve a
single retrospective pass: which of these decisions are tactical
(remove at v1.0) vs durable (keep)? The dead HTTP fallbacks in
`connectVault` / `computeSnippet` and the once-per-session
explainer flag are both tactical; the welcome notice + masked-
token settings are durable. A 30-min audit before tagging v1.0
would prevent the tactical bits from solidifying into permanent
cruft.
