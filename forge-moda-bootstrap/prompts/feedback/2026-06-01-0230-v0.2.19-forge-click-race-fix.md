## 4. JS-side production wiring

`src/main.ts:forgeSnippet` line ~1244. Located the existing
`getGenerateInventory` call via `grep`. Pre-flight sync added
immediately before the inventory call:

```typescript
const host = await pyodideHost.getInstance();

// v0.2.19: pre-flight disk→MEMFS sync. Best-effort: if the file
// can't be located or read, fall through to the preflight inventory
// call which still refreshes from whatever MEMFS currently contains.
try {
  const file = this.app.vault.getAbstractFileByPath(`${snippetId}.md`);
  if (file instanceof TFile) {
    const freshContent = await this.app.vault.read(file);
    await host.syncUserVaultFile(file.path, freshContent);
  }
} catch (e) {
  console.warn('Forge: pre-flight sync failed before /generate', e);
}

const inv = await host.preflightThenInventory(snippetId);
```

The TFile-lookup-by-path uses `<snippetId>.md`. For unqualified
snippet ids (user-vault authoring), that's the right resolution.
For qualified ids like `forge-moda/setup`, the path
`forge-moda/setup.md` should also resolve via Obsidian's file index.

Best-effort try/catch means: if Obsidian's file index doesn't have
the file (rare — but possible for snippets created externally and
not yet indexed), the sync skips and the preflight helper falls
back to whatever MEMFS contains. Non-fatal; logs a warn.

`getGenerateInventory(snippetId)` → `preflightThenInventory(snippetId)`
swap is the actual /generate-path change. The hook from v0.2.18
remains as belt-and-suspenders.

## 12. One observation

This release closes the second-order race that the v0.2.18 fix
itself introduced — async hooks fixing async hooks. The v0.2.19
fix uses a different shape entirely: **synchronous pre-flight
inside the user-action handler**, not a deferred event listener.

The pattern: **when a Forge-click depends on disk state being
current, the click handler itself synchronously reconciles disk →
MEMFS → registry → inventory before invoking the network call.**
No reliance on prior async work having completed. The
`vault.on('modify')` hook stays for the case where the user types
but never clicks Forge (so a future click 30 seconds later sees
fresh inventory), but the click handler doesn't NEED the hook to
have fired.

Worth flagging in the v1.0 retrospective alongside the prior TDD
rider: **user-action handlers should be synchronous-by-default for
state reconciliation. Background hooks are belt-and-suspenders,
not the primary path.** Adopting this as a convention would
prevent the v0.2.18-style "async hook fix introduces a race"
trap in future work.
