## 10. Deviations

**One minor.** Pre-emptively `rm -rf .forge/` from
`~/projects/forge-music/` before commit — the test run had written
live snapshots there. Without removal the working tree would have
shown an untracked `.forge/` that the engine's `_capture_edge`
would re-emit on every test run. forge-moda's published vault DOES
track a couple of `.forge/edges/` files (frozen snapshots
deliberately committed), so the convention isn't blanket-ignore
the directory — but live test-run snapshots aren't release
content. The cleaner long-term fix is a `.gitignore` rule in
forge-music for live edges only, but the prompt explicitly
restricts to migration scope; flagged here for follow-up.

## 11. One observation

**The "blues set" the prompt envisioned doesn't exist yet** —
sandbox has 2 snippets, not the 6-8 the prompt enumerated. The

Separately: the `test_form_serializes_to_musicxml` assertion checks
for `<score-partwise>` because music21's default exporter emits
partwise MusicXML 3.x. If the engine's `_try_serialize_music21`
ever switches to `<score-timewise>` mode (the alternative MusicXML
root), this test would need updating. Today neither the engine nor
any consumer cares, so a one-line note is sufficient.
