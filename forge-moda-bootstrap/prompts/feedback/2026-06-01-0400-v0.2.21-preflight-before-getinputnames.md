## 2. Fix shape

Single-line addition at the top of `_forge_get_input_names`:

```python
relpath = f"/bundle/user-vault/{snippet_id}.md"
try:
    _forge_registry.refresh_file(relpath)
except Exception:
    pass
```

Matches the pattern from `_forge_preflight_then_inventory` (v0.2.19)
exactly. The helper is now race-proof from any caller's
perspective.

**Chose option (b)** from prompt §2 — bake refresh into the helper
rather than wrap with explicit JS-side preflight. Rationale: makes
the helper self-sufficient, eliminates a class of "caller forgot
to pre-flight" bugs entirely. The JS-side `getInputNames` call
site in `main.ts:runSnippet` stays unchanged.

## 3. JS-side wiring left as-is

`main.ts:runSnippet`'s `await host.getInputNames(snippetId)` call
needs no change. The helper handles the race internally. No
additional `syncUserVaultFile` pre-flight added because the helper-
side refresh is sufficient for the race shape demonstrated in
tests 1, 3, 4.

If user smoke surfaces a case where the JS-side needs to also push
fresh disk content (the bug shape where MEMFS itself is stale —
e.g., Obsidian wrote to disk but the vault.modify event hasn't
propagated to MEMFS), a JS-side `syncUserVaultFile` would be
needed. The v0.2.19 `generate()` path already does this; runSnippet
could add the same pattern if the smoke surfaces such a case.
Logging this as a v0.2.22 candidate; not in current scope.

## 11. One observation — third "production race that suite tests miss" pattern

Per the prompt's §6 observation. Three race-fix releases in this
arc:

| Release | Fix surface | Race shape |
| --- | --- | --- |
| v0.2.18 | `vault.on('modify')` hook | async event vs sync Forge-click |
| v0.2.19 | `preflightThenInventory` in `generate()` | sync pre-flight on hot path |
| v0.2.21 | refresh inside `_forge_get_input_names` | helper-self-refresh |

**Systemic finding**: any function that reads from
`_forge_resolver.resolve(...)` has a stale-data risk window unless
it refreshes first. The clean pattern is now established —
**engine helpers should self-refresh-then-resolve, not rely on
JS-side pre-flight.**

Candidate audit sites for the v1.0 retrospective:

- `_forge_get_generate_inventory` — currently wrapped by
  `_forge_preflight_then_inventory` in v0.2.19's
  `preflightThenInventory` host method. The wrap pattern works
  but adds an indirection; the helper-self-refresh pattern from
  v0.2.21 is cleaner.
- `_forge_list_snippets` — used by the v0.2.6 connectVault path.
  Lower stakes (used at vault-connect time, not on every Forge-
  click), but worth confirming the same race-risk analysis.
- `_forge_compute` (line 481 in pyodide-host.ts) — also calls
  `_forge_resolver.resolve(snippet_id)`. The v0.2.16 diagnostic
  observed body=Nch matching the post-edit state, suggesting it
  DOES see fresh content somehow — possibly because writeGeneratedCode's
  syncUserVaultFile already pushed content + refreshed before
  compute fires. Worth confirming.

Worth a v1.0 prompt to consolidate: "Every Python helper that
calls `_forge_resolver.resolve` MUST first call
`_forge_registry.refresh_file(target)` for the snippet path."
Then audit + apply the pattern uniformly. Until then, each helper
has its own ad-hoc race-fix history.
