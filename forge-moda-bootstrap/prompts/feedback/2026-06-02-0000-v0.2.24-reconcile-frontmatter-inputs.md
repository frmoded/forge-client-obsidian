## 2. JS-side wiring

`main.ts:writeGeneratedCode` loop adds one new step. Pre-v0.2.24
sequence was:

```
1. read current content
2. replace Python section with new code
3. app.vault.modify (write to disk)
4. host.syncUserVaultFile (push to MEMFS)
5. syncDependencies (engine B7)
```

Post-v0.2.24:

```
1. read current content
2. replace Python section with new code
3. app.vault.modify (write to disk)
4. host.syncUserVaultFile (push to MEMFS)
5. reconcileFrontmatterInputs (write frontmatter inputs:)  ŌåÉ NEW
6. syncDependencies (engine B7)
```

The reconciliation call is idempotent (returns `'no-op'` if current
matches inferred), so re-running Forge with no changes doesn't
churn the file. The Obsidian `app.fileManager.processFrontMatter`
serializer is the same one Obsidian uses for in-place
frontmatter edits, so the YAML output blends with whatever the
user authored elsewhere in the frontmatter.

The glue method is fully encapsulated as a private method on the
plugin class:

```typescript
private async reconcileFrontmatterInputs(file: TFile, snippetId: string): Promise<void> {
  await reconcileInputs(snippetId, {
    getInferredInputs: async (id) => {
      const hostManager = getPyodideHost();
      if (!hostManager) throw new Error('Pyodide host not wired');
      const host = await hostManager.getInstance();
      return host.getInputNames(id);
    },
    readCurrentInputs: () => {
      const fm = this.app.metadataCache.getFileCache(file)?.frontmatter;
      return Array.isArray(fm?.inputs)
        ? (fm!.inputs as unknown[]).map(String)
        : [];
    },
    writeInputs: async (next) => {
      await this.app.fileManager.processFrontMatter(file, (fm: any) => {
        fm.inputs = next;
      });
    },
  });
}
```

### Comment 3 ŌĆö Idempotence check is the real load-bearing test

Of the 5 cases, the most valuable for the long arc is case 3
(matching current + inferred is a no-op). Without it, a future
refactor that "fixes" the comparison logic to be over-eager (say,
always writing the inferred list even when it matches current)
would silently churn every file on every /generate. The user would
see Obsidian's "file modified" indicator firing on every Forge
click. Catching that at suite time is the whole reason the
pure-core extraction matters.

Worth noting for future bug-fix prompts: **the most valuable case
in a TDD set is often the "no-op should remain no-op" assertion,
not the "fix the broken case" assertion.** The broken case forces
the fix to land; the no-op case prevents the fix from becoming
its own bug.
