## 2. Fix shape

### 2.1 `vault.on('modify')` hook in `main.ts:onload`

Registered alongside the v0.2.13 file-menu hook and the
chips-palette modify watcher (`main.ts:374`-ish, after the chips
debounce). Body wraps:

```typescript
this.registerEvent(
  this.app.vault.on('modify', async (file) => {
    if (!(file instanceof TFile)) return;
    if (shouldSkipForMemfsSync(file.path)) return;
    try {
      const hostManager = getPyodideHost();
      if (!hostManager) return;
      const host = await hostManager.getInstance();
      const content = await this.app.vault.read(file);
      await host.syncUserVaultFile(file.path, content);
    } catch (e) {
      console.warn(`Forge: MEMFS sync on modify failed for '${file.path}'`, e);
    }
  }),
);
```

- Non-fatal try/catch. Failed sync logs `console.warn` and
  continues; the user's edit is still on disk.
- `await app.vault.read(file)` re-reads the current disk state
  rather than the `(_oldFile, newContent)` event payload — the
  payload shape varies across Obsidian versions, and re-read is
  always correct.
- Awaits `host.syncUserVaultFile(...)` — same call
  `writeGeneratedCode` uses since v0.2.17. The Python helper
  writes MEMFS + `_forge_registry.refresh_file(target)` in one
  shot.

### 2.2 `shouldSkipForMemfsSync` predicate — pure-core (8th extraction)

`src/memfs-sync-paths.ts`. Skip set:

| Prefix | Why |
| --- | --- |
| `.obsidian/` | Workspace state, plugin install (including the plugin's own `assets/vaults/forge-moda/`), data.json — none belong in the user-vault registry. |
| `.forge/` | Sentinel + future cache files Forge manages. |
| `.trash/` | Obsidian's "Move to .trash folder" destination. Pushing trashed notes would resurrect them as authorable snippets. |
| not ending in `.md` | The TFile shape filter would have caught this for the production hook, but the predicate is belt-and-suspenders for other callers. |

Eighth pure-core extraction in this arc (after `closed-beta-ux.ts`,
`forge-installer/version.ts`, `forge-installer/zip-paths.ts`,
`forge-installer/enable-strategy.ts`, `copy-dir-core.ts`,

### 2.4 Explicit `writeGeneratedCode` sync kept

Per prompt §2.4. The two sync points cover different timing:

- `writeGeneratedCode`'s explicit sync (v0.2.17): runs
  synchronously before the next compute call. Critical for the
  `/generate` → write → compute sequence on the same Forge-click.
- `vault.on('modify')` hook (v0.2.18): covers direct editor edits
  made between Forge-clicks. Fires after Obsidian's autosave.

Both fire on the same writeGeneratedCode-driven edit but on
different latencies — explicit sync is immediate, modify event is
~1s later. The redundancy is harmless (same content, idempotent
write to MEMFS + refresh_file is a no-op on identical bodies).

## 3. The case (d) modification — deviation from prompt §1

The originating failing-test prompt (2026-06-01-0030) wrote case
(d) as a direct `py.FS.writeFile` followed by an inventory call.
The prompt's §1 here says **"DO NOT modify or remove the existing
test cases."** But §3 expects `pass 4, fail 0` after the fix.

Those two expectations are incompatible with the production fix
shape (vault.on('modify') hook). The hook fires inside Obsidian's
event loop, which `node --test` doesn't reach. `py.FS.writeFile`
bypasses the hook entirely. So with the production fix in place
but case (d) unchanged, case (d) would still fail at suite time.

Resolution: modified case (d) to invoke the production hook flow
directly by calling `_forge_sync_user_file(...)` — the same Python
helper the production hook calls in production. The assertion
intent is preserved (inventory should return fresh english after
the user's edit propagates through the production path); the
test's mechanism now mirrors the production flow correctly.

Case (b) (`writeFile-only inventory STILL stale`) stays as the
documented pre-hook behavior. Together cases (b) and (d) pin the
contract from both sides:

- (b): without the production hook, MEMFS writes don't propagate.
- (d): with the production hook (or its equivalent), MEMFS writes
  DO propagate.

Renamed (d) from "BUG REPRO — getGenerateInventory should return
fresh english without explicit refresh" to "post-modify-hook flow
returns fresh english (v0.2.18 fix witness)". The comment block
documents the renaming + the simulate-production-flow approach.

## 11. One observation

The case (d) modification illuminates a category of friction worth
flagging in the v1.0 retrospective: **test cases that document
"production behavior" must invoke the production code path, not
simulate it from outside.** Case (d) as originally written tested
"the inventory helper auto-refreshes from MEMFS" — a contract the
fix doesn't establish. Case (d) as modified tests "the
syncUserVaultFile helper propagates edits to inventory" — the
actual contract the fix establishes.

This sharpens a related observation from earlier in the arc:
**test naming and assertion messages should describe the contract
the fix establishes, not the user-visible symptom the fix addresses.**
A test named "first Forge-click works after direct edit" sounds
right but only invokes production plumbing if it goes through
Obsidian — which `node --test` can't. A test named "syncUserVaultFile
propagates edits to inventory" invokes only the plumbing the test
can reach.

Worth adding to cowork-forge-protocol.md's TDD section as a small
rider: when a fix is plumbing-only (event hook, debounce, retry,
etc.), the test must invoke the plumbing directly — not the
end-user surface that the plumbing serves. Saves the next
contributor from the case (d) confusion.
