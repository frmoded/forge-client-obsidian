# V1 Phase 2 — iframe routes through plugin Pyodide; ships clean

## TL;DR

V1 Phase 2 shipped after a smoke-iteration cycle. The iframe's
HTTP adapter became a postMessage adapter; the iframe loads from
the plugin's bundled `assets/iframe/` (`app://` URL scheme) instead
of `localhost:5173`; the plugin's `moda-view.ts` dispatches
`engine-request` ops (`moda-init`, `moda-compute`, `moda-click`,
generic `compute`) through the Phase 1 `PyodideHost` and replies
with `engine-response`. **Five Obsidian-context bugs that the
spike couldn't surface were caught and fixed during smoke;
acceptance criteria met.**

## Path from Phase 1 → Phase 2 ship

This wasn't a single drain-prompts cycle. It was iterative smoke
debugging across one session: I shipped Phase 1, the user did
live Obsidian smoke, hit each new failure mode, I diagnosed +
patched, the user re-smoked. Five bugs total surfaced this way,
each one isolated and fixed without breaking the next round.

## Bug 1 — iframe URL still `localhost:5173`

**Surfaced by:** Step 2 of the smoke (iframe pane opens empty in
Bluh, dev tools showed it trying to fetch from the Vite dev server).

**Root cause:** The plugin's `moda-view.ts` hardcoded
`MODA_CLIENT_URL = 'http://localhost:5173'`. V1's "no localhost"
goal required serving the iframe from the plugin's bundled assets.

**Fix:** Two-commit chain. (1) `forge-moda-client/63c66f2` adopts
`vite-plugin-singlefile` so the production build emits a single
`index.html` (~210 KB) with all JS/CSS inlined, eliminating the
split-asset references that broke under `app://`. `base: './'`
makes internal asset references work under any URL scheme. The
build's `outDir` writes directly into
`forge-client-obsidian/assets/iframe/`. (2) `forge-client-obsidian/
da08c4e` rebuilds with the new bundle and switches the moda-view's
iframe `src` to `pluginAssetUrl('iframe/index.html')`.

## Bug 2 — `node:url` CORS block

**Surfaced by:** Console error
`Access to script at 'node:url' from origin 'app://obsidian.md'
has been blocked by CORS policy`.

**Root cause:** Pyodide's ESM build detects Node by checking
`typeof process.versions.node == "string" && !process.browser`.
Obsidian's Electron renderer has `process.versions.node` set
(Node integration enabled) but `process.browser` undefined, so
Pyodide takes the Node path and tries `await import('node:url')`.
The renderer's CORS layer blocks `node:` URLs ("only chrome,
chrome-extension, data, http, https schemes are supported").

**Fix:** `6e5bc1e` — set `process.browser = true` for the duration
of the dynamic import + `loadPyodide()`. `IN_NODE` is computed
once at module evaluation, so the override only needs to be live
during the import. Restored in `finally` so unrelated code that
inspects `process.browser` isn't affected.

This was the load-bearing-untested assumption I called out in V1
Phase 1's feedback ("CSP / WASM verification: not empirically
verified — if the user's smoke reveals a CSP block, route the
follow-up to questions/"). It DID reveal a block, but the
surgical shim worked instead of needing an architecture flip.

## Bug 3 — cache-buster query string in indexURL

**Surfaced by:** Pyodide loads but every wheel/lockfile/wasm
fetch 404s with URLs like
`assets/pyodide/?1779562984640/pyodide-lock.json` — the
cache-buster appearing between the directory and the filename.

**Root cause:** Obsidian's `app.vault.adapter.getResourcePath()`
appends a `?<timestamp>` for cache invalidation. Fine for one-shot
fetches, but Pyodide's `loadPyodide({indexURL})` concatenates
indexURL + filename for its lockfile, stdlib, and wasm. The
resulting URL has the query string in the middle of the path;
Obsidian's URL resolver treats everything after `?` as the query,
so the path the resolver sees is just `assets/pyodide/` (a
directory).

**Fix:** `a9b7dcd` — strip `?<...>` from URLs returned by
`pluginAssetUrl()` before passing to Pyodide. Path-side resolution
works fine without the cache-buster for our static read-only
plugin assets.

## Bug 4 — wheels in wrong directory

**Surfaced by:** Pyodide loaded but `loadPackage(["numpy",
"pyyaml", "micropip"])` 404'd on every wheel. The error showed
Pyodide looking for them at `assets/pyodide/numpy-…whl`, but
they were in `assets/wheels/`.

**Root cause:** Pyodide's `loadPackage` resolves wheel URLs
relative to `indexURL` (i.e., `assets/pyodide/`). The lockfile
gives wheel filenames without paths; Pyodide expects them
alongside its core. Our split into separate `pyodide/` and
`wheels/` directories was an organizational choice that didn't
match Pyodide's expected layout.

**Fix:** `9eeeec9` — updated `setup-assets.mjs` to place all
wheels in `assets/pyodide/` (matching Pyodide's "full"
distribution layout where pyodide.asm.wasm + python_stdlib.zip +
every wheel live together). Also live-moved bluh's wheels for
the running smoke. `.gitignore` updated to drop the now-unused
`assets/wheels/` entry.

## Bug 5 — postMessage clone failure + canvas not redrawing on featured-run

**Surfaced by:** Step 9 (click "Run simulation"). Console showed
`moda featured-run failed: Failed to execute 'postMessage' on
'Window': [object Object] could not be cloned.` plus a cascading
`view.appendError is not a function`. Canvas didn't redraw.

**Root cause #1:** The Pyodide path's `_forge_compute` returned
the raw Python `ParticleState` dataclass via `toJs()`. That comes
through as a Map/PyProxy of numpy arrays — non-cloneable. The
HTTP `/compute` endpoint always called `serialize_result()` first
to produce the wire envelope
`{type: "moda_sim_state", content: {tick, particles: [...]}}` —
pure JSON. Our Pyodide path skipped that step. Result:
- Iframe's check `result.type === "moda_sim_state"` was false
  (no type tag), canvas redraw branch never ran.
- The subsequent relay `postMessage` to `window.parent` threw on
  the un-cloneable Map.

**Root cause #2 (cascade):** The iframe's error-relay tried to
forward the failure to Forge Output. The plugin's
`relayComputeResult` got a `DeferredView` placeholder from
`getOrOpenOutputView`'s existing-leaf branch. Obsidian lazy-loads
leaves from prior sessions; `leaf.view` is a placeholder without
`ForgeOutputView`'s subclass methods until activated.

**Fix:** `eb6ed45` — two changes in one commit:
- `_forge_compute` imports and calls `serialize_result(raw, snip)`
  before returning, mirroring the HTTP endpoint. Result is now a
  pure-JSON wire envelope that survives postMessage AND that the
  iframe's renderer dispatches on correctly.
- `getOrOpenOutputView` detects the deferred case (view isn't an
  instanceof `ForgeOutputView`) and re-asserts `setViewState` to
  materialize the real view. Plus a `typeof` safety net in
  `relayComputeResult` that inlines the error as a regular
  `append` if `appendError` is still unavailable.

## Performance reality check

Spike-predicted moda budget (Node.js Pyodide, Python only):
~26ms/tick → 38fps.

Real Obsidian (full loop: Python + canvas + 2× postMessage):
~72-75ms/tick → ~14fps wall-clock.

The ~50ms delta is canvas redraw (React reconciliation + 500+
`fillStyle/arc/fill` calls per frame) + two structured-clone
round-trips per tick (engine-request out, engine-response back)
+ Obsidian event-loop overhead. The Python compute itself is
unchanged; the iframe-vs-plugin architecture has overhead the
spike didn't measure.

Still above the floor where it'd feel sluggish — the live
simulator paints smoothly per the user's smoke. The featured-
button one-shot (300-tick simulation) finishes in ~8-15 seconds
in real Obsidian (vs ~8s in spike).

## Architecture validated

V1 Phase 2 confirms architecture A (plugin hosts Pyodide). The
spike's verdict translated to the real Obsidian context with
five surgical fixes — no architecture flip required, no fallback
to architecture B (iframe hosts Pyodide), no engine changes.

The shape of the postMessage protocol is also confirmed:
`engine-request` (with `op`, `args`, `vault_name`, `request_id`)
+ `engine-response` (with `request_id`, `ok`, `result`, `error`)
+ correlation via UUID-per-call lets the live 30Hz loop race
against user canvas clicks without confusion.

## One observation

The biggest hidden cost in V1 Phase 2 was **the gap between
"Pyodide works in Node" (spike) and "Pyodide works in Obsidian"
(real)**. The spike couldn't surface: Electron's Node integration
fooling `IN_NODE`, Obsidian's cache-buster query string, the
deferred-view placeholder, the `getResourcePath` URL scheme.
Each of those is a real bridge between environments — Node vs.
Electron renderer, browser fetch vs. Obsidian's `app://`
resolver, dev-tools-clean structured clone vs. real PyProxy
leakage.

Worth flagging for any future spike-vs-ship cycle: budget time
for ~5 Obsidian-context interface mismatches even after a clean
spike. They're not failures of the spike's verdict; they're the
adapter layer the spike didn't have to write.
