## 2. Fix shape rationale

- `inputs?: Record<string, unknown>` with default `{}` keeps
  existing call sites compatible (moda fast-path, iframe protocol).
- `.to_py()` is the canonical Pyodide cast from JS object to
  Python dict. The prompt §2.1 suggested
  `dict(_forge_inputs_in or {})`, but that fails: JsProxy isn't
  directly dict-iterable in Pyodide. Confirmed by case (1) of
  the test which failed at first with this exact cast error
  before I switched to `.to_py()`. See §7 deviation.
- The `_forge_inputs_in.to_py() if _forge_inputs_in else {}`
  guard handles the default-empty-dict case where JS passes
  `{}` (truthy JS object but produces an empty Python dict).

## 7. Deviations + observations

### Deviation: `.to_py()` instead of prompt's `dict(_forge_inputs_in or {})`

The prompt §2.1 suggested `dict(_forge_inputs_in or {})` to coerce
the JS object into a Python dict. That doesn't work — Pyodide's
`JsProxy` (what a JS object becomes when passed through
`globals.set`) isn't directly dict-iterable. Discovered when
test case (1) initially failed with:

```
TypeError: 'pyodide.ffi.JsProxy' object is not iterable
```

The canonical Pyodide pattern is `.to_py()`, which recursively
converts JS values to Python types. Updated both the production
impl AND the test fixture to use `.to_py()`. Documented in the
runPython block as a code comment.

The prompt §7 explicitly invited this kind of deviation
("Whether `dict(_forge_inputs_in or {})` needed any other
coercion (e.g., `to_py()` conversion for nested objects).").

### Deviation: test fixture rename

The mirror helper was named `computeViaEngine_v0_2_21_PreFix`
when first written (anticipating it'd diverge from the post-fix
production impl). After the fix landed, it's an exact mirror,
so renamed to `computeViaEngine_PreFix` (the "Pre" prefix retained
because production lives in obsidian-imports-bound code that
node --test can't reach; the helper is the suite-side substitute).
Per the recently-updated TDD rider's permission to refactor test
mechanism without breaking assertion intent.

### Observation: latent-since-v0.2.6

This bug existed for ~16 plugin releases (v0.2.6 introduced the
Pyodide compute path; v0.2.22 fixes the inputs-threading at that
boundary). It survived because no production snippet routed
user-supplied modal kwargs through the JS-side boundary until
music week's first user-authored snippet with parameters
(Greet's `def compute(context, name)`).

The race-chase arc (v0.2.17 → v0.2.21) found real adjacent bugs
— async hook timing, MEMFS staleness, registry refresh — but
the proximal cause stayed masked because the suite tests never
exercised computeViaEngine with non-empty kwargs.

## 11. One observation for the v1.0 retrospective

**This bug is the canonical example of why the new
"push every assertion to suite up to UI boundary" rule matters.**

The race-chase arc consumed five releases (v0.2.17 → v0.2.21)
looking for the wrong root cause. A single suite-level test
that called `host.computeViaEngine('any-snippet', [], {x: 1})`
and asserted `x` reached Python would have caught the latent
v0.2.6 bug at suite-run time, immediately.

Recommendation for v1.0 retrospective:
**every JS→Python boundary helper should have at least one test
that round-trips a non-trivial argument through Python and
asserts it arrived intact.** The boundary itself is a class of
bug that unit tests of either side miss.

Candidate audit sites for v1.0:
- `computeViaEngine` (this fix)
- `syncUserVaultFile` — verify the body content survives the
  round-trip + ends up in MEMFS
- `getInputNames` — verify the snippet_id is correctly resolved
  (also covered by v0.2.21's race tests)
- `preflightThenInventory` — verify the snippet_id resolves
- `getConnectInventory` — verify the vault structure crosses
  the bridge correctly

For each boundary helper, the test pattern is identical: pass
a non-default argument, assert it reached Python (or that the
JS-side return value reflects it). The test cost is small per
helper; the cumulative protection against "silent argument
drop" bugs is large.
