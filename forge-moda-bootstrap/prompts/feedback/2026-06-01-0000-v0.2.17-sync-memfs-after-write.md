## 2. Fix shape

### 2.1 SnippetRegistry refresh API chosen

Added a **new method** `refresh_file` to `SnippetRegistry`.
Existing methods (`scan`, `get_in_vault`, `get_bare`) all walk the
filesystem or return cached entries; none does the per-file
re-parse-and-update we need. Adding `refresh_file` is the
least-invasive option per prompt §2.1:

```python
def refresh_file(self, filepath: str, vault_name: str = AUTHORING_VAULT,
                 source: str = "authoring") -> Optional[str]:
    """v0.2.17: re-index a single file from disk and update the
    cached entry. ..."""
    vault_path = None
    if vault_name in self._vaults:
        for entry in self._vaults[vault_name].values():
            vault_path = entry.get("vault_path")
            if vault_path:
                break
    if not vault_path:
        vault_path = os.path.dirname(filepath)
    self._vaults.setdefault(vault_name, {})
    return self._index_authoring_file(filepath, vault_name, source, vault_path)
```

`vault_path` inference is the only non-trivial part: V1 single-
vault model means `vault_name="authoring"` always has at least one
entry once the initial scan completes; the inference walks until
the first non-empty `vault_path` field. The parent-dir fallback
covers cold-cache cases (registry never scanned — edge case but
defensive).

Returns same error-string-or-None shape as `scan()`'s
`self.errors` entries so callers can treat the two consistently.

Mirrored to engine source at
`forge/forge/core/snippet_registry.py` (commit `3a67e82` on
`forge/main`). Bundled copy at
`forge-client-obsidian/assets/engine/forge/core/snippet_registry.py`
is byte-identical for the new method.

### 2.2 Python sync helper

`_forge_sync_user_file(relpath, new_body)` added to the inlined

### 2.3 TS surface

`PyodideHostInstance.syncUserVaultFile(relPath, content)`. Sets
the two globals and runs the Python helper. Same shape as
`getConnectInventory`'s tuple-args pattern.

### 2.4 Wired into writeGeneratedCode

`main.ts:writeGeneratedCode` — after each `app.vault.modify`:

```typescript
try {
  const hostManager = getPyodideHost();
  if (hostManager) {
    const host = await hostManager.getInstance();
    await host.syncUserVaultFile(file.path, newContent);
  }
} catch (e) {
  console.warn(`Forge: MEMFS sync after write failed for '${id}'`, e);
}
```

Non-fatal try/catch — host-not-wired path logs + continues
(defensive against the pre-v0.2.6 HTTP-compute fallback). `file.path`
is vault-relative, matches MEMFS layout.

### 2.5 §2.4 (editor-change debounced hook) — deferred

Per prompt §2.4 guidance: "If this is a 30-minute add, include it.
If it's a rabbit hole, defer to v0.2.18 and document." Direct
editor-change sync is deferred. Rationale:

- The greet-bug repro path is Forge-click → /generate →
  writeGeneratedCode → compute. That path is fully covered by
  §2.3.
- Direct editor edits between Forge-clicks would still need
  sync, but the next Forge-click on the same snippet triggers
  /generate → writeGeneratedCode which IS hooked. So the gap
  only matters for "edit then immediately Forge-click without
  re-running /generate" — uncommon enough to defer.
- A debounced `vault.on('modify')` adds ~30 lines including the
  debounce logic; defer to v0.2.18 if smoke confirms the user
  still hits a related staleness path.

## 10. One observation

TDD discipline keeps paying off. v0.2.16 used it to refute
hypothesis 1a (parser bug) in ~700ms — saving a wasted release.
v0.2.17 uses it to LOCK IN the bug at suite time before fixing —
case (c) in `memfs-staleness.test.ts` will catch any future
regression of "disk write doesn't propagate to MEMFS" in any
form, even if the surrounding plumbing changes.

The pattern across both v0.2.16 and v0.2.17 investigations: write
the test → run it → either it refutes the hypothesis (pivot) or it
locks in the bug (apply fix with confidence). Both outcomes are
useful; both are cheap.

**Recommendation for v1.0 retrospective**: the
cowork-forge-protocol's "Bug-investigation prompts use TDD
discipline" §6 should be promoted from "discipline to follow" to
"hard rule for any bug-related prompt." Two consecutive drains
have validated it; the cost is small (~30 minutes of test-writing
front-loaded) and the payoff is large (no wasted releases, every
investigation produces a permanent regression test).
