## 6. Protocol comments for driver

The user explicitly asked the feedback to include "all protocol
comments for driver" — i.e., my running observations on how the
updated cc-prompt-queue.md and cowork-forge-protocol.md shaped
this drain. Six observations:

### Comment 1 — §80 (test-fixture drift rider) is load-bearing and prevented a hidden re-bug

The new rider was added between my v0.2.22 drain and this one,
specifically motivated by v0.2.22's fixture-lie failure mode. The
v0.2.23 prompt is the first to test the rider against a real bug
shape, and the rider performed as intended: I restructured the
fixture to dynamically load production code before writing any
fix, which surfaced one MORE failing case (case 1) than the
v0.2.22 fixture had reported. Without the dynamic load, I would
have applied the prompt's described fix and the suite would have
passed against a still-buggy fixture — exactly the v0.2.22 trap.

**Recommend the rider stays exactly as written.** The mechanism
(extract production block via marker regex, run it directly) is
~30 lines and zero ongoing maintenance cost.

### Comment 2 — Backtick-in-docstring trap surfaced AGAIN (second occurrence)

v0.2.20's feedback flagged this and called it "worth remembering."
v0.2.23 hit the same trap mid-drain — I added new docstrings with
inline-code marks (` `...` `) inside the Python block, the
template literal terminated early, build failed. Fix is trivial
(single quotes) but the lesson hadn't been codified.

**Recommend adding to cc-prompt-queue §"Test-infrastructure
conventions"** (or §80 directly) a small bullet:

> **Python docstrings inside JS template literals must not use
> backticks for inline code marks.** Backticks terminate the
> outer template literal. Use single quotes for inline-code marks
> (`'inputs'`, `'{}'`) or restructure the docstring to avoid the
> need.

This is small but recurring; codifying it costs one line and
saves me a mid-drain build-error round-trip.

### Comment 3 — The §1.1-§1.5 scaffold is clean to use, but §1.3 ambiguity remains

The "fix landed" section (§1.3 per the protocol) can either:
- Cite just the commit hash + one-sentence shape, OR

I went with the second option this drain (inline code blocks for
the two changed Python functions). The first option keeps the

Either reads fine; the protocol doesn't say which. **Suggest
clarifying in §66-74 that §1.3 SHOULD include the diff inline for
the load-bearing changes** — it's a tiny addition that prevents

### Comment 4 — "Smoke after review, not in prompt" cowork rule played out cleanly this drain

### Comment 5 — The marker-based extraction is robust enough; consider a watchdog test

The `_PYTHON_BLOCK_BEGIN`/`_END` markers + regex extraction in
`bootGreet` is robust for one production block. If a second
inline Python block ever needs the same treatment (e.g., a
plugin's onload handler with its own Python prelude), the regex
pattern would need to disambiguate which block. Not a v0.2.23
problem — but worth flagging.

**Suggest adding a tiny watchdog test** that asserts the
extraction found a non-empty block. v0.2.23's fixture currently
throws if the markers aren't found; a positive assertion in some
test file ("block extracted, length > 100 chars, contains
`def _forge_compute`") would catch future regressions where
someone deletes the markers or splits the block. ~5 lines, no
maintenance burden.

## 9. One observation for the v1.0 retrospective

**Two boundary-bug surfaces, two latent-since-v0.2.6 bugs, two
fixture-drift failure modes.**

- v0.2.22: JS-side `computeViaEngine` dropped inputs. Fixture lied
  by accident; tests passed against fixture; production broke.
- v0.2.23: Python-side `_forge_compute` + `_forge_run_snippet`
  dropped inputs. Fixture lied by accident (the v0.2.22 fix
  applied to the fixture but not to production); tests passed
  against fixture; production stayed broken.

Both bugs would have been caught at v0.2.6 by a single test that
called `host.computeViaEngine('any-snippet', [], {x: 1})` and
asserted `x` reached Python.

The new "dynamic-load fixture" rider catches the v0.2.22-style
trap (suite mirror != production). The "push every assertion to
suite up to UI boundary" rule (§126-131) catches the v0.2.6-style
trap (no test covers the boundary at all). Both rules together
close the trap; either alone leaves a hole.

**Recommend: v1.0 audit task to apply the §126-131 rule
systematically to every JS↔Python boundary helper** —
`computeViaEngine`, `syncUserVaultFile`, `getInputNames`,
`preflightThenInventory`, `getConnectInventory`, etc. Each
gets one suite test calling it with a non-default argument and
asserting the argument reached Python intact. Per-helper cost is
small (~15 lines); cumulative protection against silent
argument-drop bugs across the entire boundary is large.
