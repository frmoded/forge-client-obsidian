## 2. First-run extracts bundled forge-moda

`runFirstRunCheck` extended with `ensureBundledForgeModa(app)`.
Behavior:

- **Independent of the sentinel** — fires on every plugin load
  rather than only first-run. Covers both fresh vaults (sentinel
  absent + no `forge-moda/`) AND upgrade-without-content vaults
  (sentinel present from a pre-bundled version + no `forge-moda/`
  yet).
- **Skips if `<vault>/forge-moda/` exists** — protects user
  edits + avoids re-copying on every plugin load.
- **Source**: `.obsidian/plugins/forge-client-obsidian/assets/vaults/forge-moda/`.
  If missing (build glitch), warns + returns; doesn't throw.
- **Non-fatal try/catch** wrapping the whole extension so plugin
  load can't abort even on adapter weirdness.

`copyDirRecursive` lives in `src/copy-dir-core.ts` (new pure-core
file). `welcome.ts` re-exports it for backwards compat. The
extraction was forced by the obsidian-import boundary at
`node --test` time (same boundary that forced `closed-beta-ux.ts`
and `forge-installer/version.ts` / `zip-paths.ts` splits — see §4
and §9).

## 11. One observation

The `obsidian`-package-as-types-only-stub boundary has now forced
five pure-core extractions in this arc: `closed-beta-ux.ts`,
`forge-installer/version.ts`, `forge-installer/zip-paths.ts`,
`forge-installer/enable-strategy.ts`, and now
`copy-dir-core.ts`. The convention is clear: any helper whose
logic is non-trivial enough to test should NOT live in a file
that imports obsidian at the top.

Worth adopting as an explicit rule in the v1.0 retrospective:
"new logic that needs unit testing goes in a pure-core file;
obsidian-coupled glue files re-export it." The five existing
splits already validate the pattern — it pays back the small
file-count cost in test coverage and survives Obsidian API drift
naturally.
