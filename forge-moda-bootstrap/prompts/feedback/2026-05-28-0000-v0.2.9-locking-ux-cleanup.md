### 1.1 Command-palette entry

Picked **single toggle** (not two switch-to-X commands). Rationale:

- Matches the toolbar button's semantics exactly — same callback,
  same affordance just surfaced in a second place.
- The Notice fired inside `toggleEditModeForFile` already announces
  the new mode (`Forge: {basename} → Python mode` /
  `→ English mode`), so direction is never ambiguous post-invocation.
- Two switch-to-X variants would either both surface in every Cmd+P
  search (noise) or each need conditional gating on the snippet's
  current mode (state-tied palette filtering, more code surface
  for a tiny ergonomic gain).

## 4. forge-transpile drift-check script

### 4.1 What "drift" means

The vendored `build_user_prompt` in
`forge-transpile/anthropic_client.py` and the engine's
`_build_prompt` in `forge/forge/core/llm.py` must produce the same
string output for the same input. The two implementations differ
in surface area (engine walks a registry to materialize dep info;
vendored takes pre-materialized dep list) but the actual prompt-body
formatters — `lines = [...]` and `lines.append(...)` calls — must
stay aligned.

### 4.2 Script behavior

`forge-transpile/scripts/check-prompts-drift.sh` (76 lines, exec
mode set). Heuristic: extract function body via awk
(`def <name>(` → next top-level `def`), then filter to lines
matching `lines\s*=\s*\[` or `lines.append(`, normalize whitespace,
strip comments. Diff the resulting sequences.

Exit codes:
- `0` — formatters aligned.
- `1` — drift detected; diff printed.
- `2` — script-side extraction error (missing source file, broken
  heuristic).

Picked the **lines-only diff heuristic** over a whole-function diff
because the two implementations legitimately differ in argument
parsing (engine takes `meta, body, deps, registry`; vendored takes
pre-flattened `description, inputs, english, generation_notes,
deps`). Restricting to `lines.append(...)` calls catches actual
output-shape drift while tolerating argument-shape divergence.

### 4.3 DEPLOY.md note

Added "Pre-release drift check" subsection under "Operational

### 4.4 First-run result: drift detected — but cosmetic

```
$ bash scripts/check-prompts-drift.sh
FAIL: drift detected between ../forge/forge/core/llm.py::_build_prompt and
      anthropic_client.py::build_user_prompt

--- engine (canonical) vs +++ local (vendored) ---
5c5
<     lines.append(f"Generation notes (machine-targeted authoring hints, not part of the English facet): {generation_notes}")
---
>     lines.append(
```

**Semantically the same output.** The engine's append is a
one-liner f-string; the vendored version splits across three source
lines using string-concat (`"prefix " + generation_notes`). Both
produce the identical output string when `generation_notes` is
non-empty.

The script's line-by-line heuristic catches the syntactic form of
the call, not the semantic intent — so this is a false positive
the script can't currently distinguish. Per prompt §4.4 ("don't
auto-reconcile"), leaving as observation.

**Recommended follow-up (see §10):** either reconcile the vendored
version to mirror the engine's one-liner verbatim (3-line trivial
diff), or upgrade the script heuristic to join continuation lines
of multi-line `lines.append(...)` calls before diffing. Reconciling
is the simpler fix; would let the script return 0 on every release
from day 1.

## 10. One observation

The drift script's cosmetic false positive (§4.4) is the
exact-shape problem the script was supposed to catch — except the
script is too literal. A semantic-equivalence test (run both
implementations against the same inputs, diff outputs) would be
bulletproof but requires both Python interpreters cooperating
(one ImportError each from cross-repo imports). A middle-ground:
have the drift script use Python's `ast.parse` + a small
canonicalizer to normalize `lines.append("a" + b)` to
`lines.append(f"a{b}")` before diffing. ~30 minutes of work; would
let the script ship clean from day one and survive both kinds of
real drift (semantic change AND surface re-formatting). Logging
rather than expanding scope here.
