## 2. Python helper

`src/pyodide-host.ts` inline block. Verbatim:

```python
def _forge_get_input_names(snippet_id: str):
    import ast as _forge_ast
    snip = _forge_resolver.resolve(snippet_id)
    meta = snip.get("meta") or {}
    declared = [str(i) for i in (meta.get("inputs") or [])]

    body = snip.get("body") or ""
    try:
        code = extract_python(body)
    except Exception:
        return declared
    if not code or not code.strip():
        return declared

    try:
        tree = _forge_ast.parse(code)
    except SyntaxError:
        return declared

    sig_args = None
    for node in _forge_ast.walk(tree):
        if isinstance(node, _forge_ast.FunctionDef) and node.name == "compute":
            sig_args = (
                [a.arg for a in node.args.args]
                + [a.arg for a in node.args.kwonlyargs]
            )
            break
    if sig_args is None:
        return declared

    sig_args = [a for a in sig_args if a != "context"]

    out = list(declared)
    for a in sig_args:
        if a not in out:
            out.append(a)
    return out
```

Uses Python's stdlib `ast` (already available in Pyodide stdlib).
Walks once to find the first `def compute`; combines
`node.args.args` (positional-or-keyword) + `node.args.kwonlyargs`
(PEP 3102 keyword-only); filters `context`; unions with declared
preserving declared-first ordering.

## 3. TS surface

```typescript
async getInputNames(snippetId: string): Promise<string[]> {
  this.pyodide.globals.set('_forge_input_names_snippet_id', snippetId);
  const proxy = this.pyodide.runPython(
    `list(_forge_get_input_names(_forge_input_names_snippet_id))`,
  );
  return this._unwrap(proxy) as string[];
}
```

`list(...)` cast inside runPython ensures the Python return crosses
Pyodide's bridge as a list (already a list, but defensive against
future helper changes that return iterables).

## 4. JS-side wiring

`src/main.ts:runSnippet` body around line 1498-1518. The
frontmatter-only derivation is now the fallback, not the primary
path:

```typescript
let inputs: string[];
try {
  const hostManager = getPyodideHost();
  if (!hostManager) throw new Error('Pyodide host not wired');
  const host = await hostManager.getInstance();
  inputs = await host.getInputNames(snippetId);
} catch (e) {
  console.warn(
    `Forge: signature-inferred inputs unavailable for '${snippetId}', falling back to frontmatter`,
    e,
  );
  inputs = frontmatter
    ? (frontmatter.inputs ?? [])
    : (this.lookupInventoryInputs(snippetId) ?? []);
}
```

If the engine query fails (host not yet warm, snippet
unresolvable, etc.), behavior degrades to v0.2.19's
frontmatter-only path. `lookupInventoryInputs` stays as the
deeper fallback for empty-stub-over-builtin cases.

## 9. Deviations

- **Build failure mid-session: backticks inside the Python
  docstring.** The helper's docstring originally used markdown-
  style inline-code marks (\`context\`). Inside a JS template
  literal, the backtick terminates the literal early — esbuild
  emitted `Expected ")" but found "context"`. Fixed by swapping
  backticks for single quotes in the docstring. **Lesson worth
  remembering: Python docstrings embedded in JS template literals
  must not use backticks; use single quotes for inline-code marks.**
  All 122 tests pass and the build is green after the fix.

- **§5 prompt asked for one additional "JS-side fallback test"** —
  skipped per §5's manual-smoke alternative. The JS-side fallback
  is a 4-line try/catch around the Pyodide call; testing it would
  require shimming `getPyodideHost` (currently a module-level
  singleton). The manual smoke step §8d covers the equivalent
  scenario at the live-Obsidian level.

- **Python helper placement: inline block in pyodide-host.ts**
  per §4. Not mirrored to engine source. Helper is plugin-side
  compute UX logic, not an engine API.

- **No signature-parsing edge cases surfaced** beyond the 9 the
  prompt anticipated. Decorators, async def, and type annotations
  are all handled correctly by `ast.parse` (decorators don't
  affect FunctionDef.args; async def becomes AsyncFunctionDef
  which my walk skips — note: this is a latent gap, but
  `compute(context, ...)` is documented as sync in the engine
  contract so async compute isn't a current concern).

## 12. One observation

The `getInputNames` call adds a Pyodide round-trip (~5-10ms warm,
~1.5s cold) to the Forge-click → modal-open path. Manual smoke
will confirm whether this is perceptible. If yes, a v0.2.21 polish
could pre-warm Pyodide on plugin load (instead of lazy-on-first-
compute) so the modal-open round-trip is always warm. Logging
the candidate; not a blocker for v0.2.20 ship.

The deeper observation: **two pure-data helpers shipped in this
drain — `_forge_get_input_names` and the existing
`_forge_get_generate_inventory` — both walk the snippet registry,
parse python, and surface metadata to the UI.** They're cousins.
At 3+ helpers (count the next prompt that adds another), it'd be
worth a small `forge.core.snippet_ui_metadata` module that
encapsulates them. v1.0 audit candidate; not blocking.
