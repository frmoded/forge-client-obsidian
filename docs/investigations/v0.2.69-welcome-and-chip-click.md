## Bug 1 — Welcome.md regression at v0.2.68 on forge-music source vault

**Hypothesis** (per prompt §1.1): two separate Welcome.md write paths exist
in `src/welcome.ts`. The v0.2.66 symmetric gate fix covers the lowercase
`ensureWelcomeFiles` path but misses the older capital-W
`WELCOME_PATH = 'Welcome.md'` create path. The case difference is what
hid the asymmetry from previous reviews.

**Confirmed.** Line citations against `src/welcome.ts` as of commit `0b712fb`:

- **Line 37**: `const WELCOME_PATH = 'Welcome.md';` (capital W).
- **Lines 110-123**: `runFirstRunCheck` opens with the
  `if (!hasSentinel)` block:
  ```typescript
  if (!hasSentinel) {
    const hasWelcome = await adapter.exists(WELCOME_PATH);
    console.log('Forge: Welcome.md exists?', hasWelcome);
    if (!hasWelcome) {
      await app.vault.create(WELCOME_PATH, WELCOME_NOTE);
      console.log('Forge: created Welcome.md');
    }
    // sentinel write...
  }
  ```
  No `shouldSkipBundledExtract(sourceVaultName)` gate.
- **Line 131**: `const sourceVaultName = await detectSourceVault(adapter);`
  fires AFTER the sentinel block; the capital-W writer never sees its
  return value.
- **Lines 146-168**: `ensureWelcomeFiles` (lowercase `welcome.md` +
  `greet.md`) IS correctly gated by `shouldSkipBundledExtract`.

**No other writers.** `grep -rn "Welcome.md\|WELCOME_PATH\|app.vault.create"
src/` returns the welcome.ts site as the only writer of capital-W
`Welcome.md`. `src/forge-action.ts:916` writes lowercase `welcome.md`
from `InitializeForgeVaultWizard.applyDiff`, which only fires when the
user clicks the ribbon on a vault without a forge.toml stub — different
lifecycle from `runFirstRunCheck`, irrelevant to this bug.

**isSourceVault correctness.** `src/welcome.ts:16` declares
`KNOWN_BUNDLED_LIBRARIES = new Set(['forge-moda', 'forge-music'])`.
forge-music's `forge.toml` declares `name = "forge-music"`. The
`isSourceVault(body, KNOWN_BUNDLED_LIBRARIES)` regex at
`src/source-vault-core.ts:67` matches the bare/double-quoted/single-
quoted forms. Detection works; the bug is the missing gate, not
detection.

**Fix design** (Phase 2):

1. Move `const sourceVaultName = await detectSourceVault(adapter);`
   to BEFORE the `if (!hasSentinel)` block.
2. Extract a pure-core `shouldCreateLegacyWelcomeMd(hasSentinel,
   sourceVaultName)` helper that returns true only when
   `!hasSentinel && sourceVaultName === null`. Pure-core extraction
   in `src/welcome-legacy-gate-core.ts`.
3. Replace the literal `if (!hasWelcome)` inner branch with a guard
   on the helper's return value.
4. Sentinel write should STILL fire even for source vaults — preserves
   idempotency (no re-check on subsequent reloads).
5. Add an observable `console.log` for the skip case.

## Bug 2 — chip-click guard misfires on first-clicked file

**Hypothesis** (per prompt §1.2): `lastMarkdownView` is assigned ONLY in
the `file-open` callback. Path A install workflow re-opens Obsidian with
the last-active file already restored — file-open for that file fired
during workspace boot, BEFORE `ChipsView.registerEvent` ran. The user
clicks chip without ever switching files, fallback returns null, guard
Notice fires, no insertion.

**Confirmed.** Line citations against `src/chips-view.ts` as of commit
`0b712fb` (same v0.2.67 state):

- **Line 26**: `private lastMarkdownView: MarkdownView | null = null;`
  initialized null.
- **Lines 37-47**: file-open handler is the SOLE assignment site:
  ```typescript
  this.registerEvent(
    this.app.workspace.on('file-open', (file) => {
      void this.refresh();
      const v = this.app.workspace.getActiveViewOfType(MarkdownView);
      if (v && v.file?.path === file?.path) this.lastMarkdownView = v;
    }));
  ```
- **Lines 54-57**: `onOpen` does NOT eagerly snapshot the active
  markdown view; it only calls `refresh()`.
- **Lines 274-284**: `onChipClick` resolves
  `const view = live ?? this.lastMarkdownView;` and Notices if
  `!view?.file`.

**Workspace race.** The `v.file?.path === file?.path` strict-equality
check at line 46 is over-tight: during workspace-boot ordering quirks
the active view's file may be set before the file-open event's `file`
argument is populated, leaving `v.file?.path` non-null but unequal.
The looser check `if (v && v.file)` covers the same intent (snapshot
the most-recent view that has a file) without the equality dependency.

**v0.2.67 changed `void this.render()` → `void this.refresh()`** at the
chips-view.ts file-open handler (commit `30472c5`). That diff did NOT
touch the `lastMarkdownView` assignment logic — the bug pre-dates
v0.2.67 and is surfaced by the Path A install workflow, where Obsidian
restores last workspace state with a file already open.

**No other assignment sites.** `grep -rn "lastMarkdownView" src/`
returns 4 hits: the declaration (line 26), the file-open assignment
(line 46), the render-target fallback (line 198), and the onChipClick
fallback (line 278). All read sites use it as a fallback when the
live active view is null.

**Fix design** (Phase 2):

1. **Snapshot-on-open**: in `onOpen()`, eagerly call
   `this.app.workspace.getActiveViewOfType(MarkdownView)` and assign
   to `this.lastMarkdownView` if it has a file. Closes the
   Path-A-install hole where the plugin enables while a file is
   already restored-active.
2. **Loosen file-open assignment**: drop the strict equality check;
   any `v` with a `v.file` wins. Closes the workspace-race hole.
3. **Extract `findFallbackMarkdownView` pure-core**: takes a
   structural-adapter for workspace state + the tracked `lastSeenView`
   and walks live → tracked → most-recent leaf → first markdown
   leaf with a file → null. Test the fallback chain.
4. **Wire onChipClick to use the helper** rather than its inline
   `live ?? this.lastMarkdownView` resolution.
